# 协作方案

**触发张力**: D01 — 零资质者借助 AI Agent 介入前沿知识生产
**方案版本**: plan_generator_v0
**生成时间**: 2026-02-20

---

## 一、协作全景

P06（聋人，零专业学术背景，已建立 AI 辅助学习框架，当前博客 27 次阅读）的核心张力是：如何借助 AI Agent 工具链，跳过传统专家资质路径，直接在四个前沿领域（星球探索、长寿科技、AI 安全、个人提升）产出可分享的原创技术工件。P03（pingdior）提供入域工具链，P07（张晨曦）提供深化工程脚手架，两者在知识生产流水线上时序串联，格式天然兼容——P03 帮 P06 进门，P07 帮 P06 在门里深走。P04（Mark Jin）提供一个已验证的"零资质介入前沿研究"路径案例，以及可被 P07 论文引用的外部验证工件，并与 P07 构成互为对方命题的局部证据持有关系。整体构型的能力覆盖是"如何生产知识"的全流水线支撑，而非目标领域的内容覆盖——这是本次协作的边界。

## 二、各参与者方案

### P03（西天取经的宝盖头 / pingdior）

- **角色**: 知识入域工具链提供者，负责帮助 P06 在陌生领域快速建立初步认知地图

- **贡献**:
  - 提供 newLearning 仓库中已实现的知识树提示词（输入任意领域名称，输出该领域前 100 个核心概念及详解），适用于 P06 的四个目标领域
  - 提供一对一个人辅导提示词（基于 Bloom 两标准差效应，已实现并开源）
  - 提供 autogen 三角色沙盒演示（产品经理/开发者/代理角色串联运行，P03 亲自设计的多角色流水线，具备接口设计能力）
  - 设计"P03 工具链输出格式 → P07 工程脚手架输入格式"的接口约定（P03 有多角色流水线设计经验，可主动推动接口落地）
  - 上述工具产出已是 markdown 文件和 Python 脚本并有 GitHub commit 记录，直接符合 P07 的 /deep-read 命令输入格式，集成成本接近零

- **收益**:
  - P03 Profile 明确自述寻找"内容和运营伙伴"；P06 可持续输出结构化学习笔记和概念框架，与 P03 所需的内容侧输入类型匹配（条件：P06 的输出质量被外部验证后，互补关系才真正激活）
  - 在"个人提升"方向，P06 是 P03 的 MindRing 和 onething 产品潜在的内容贡献者和用户路径来源

- **成本**:
  - 设计和维护 P03↔P07 接口约定所需的工程时间
  - 接受"先框架后行动"的哲学约束：P03 的工具链假设学习者先建立概念地图再深入，这是 P03 所有产品的内置哲学，D01 接受这套工具即接受了这个认知起点假设

- **依赖**:
  - P03 提供入域工具链这件事本身不依赖其他参与者，但接口约定需要 P07 确认 /deep-read 命令的输入格式规范，才能完成 P03→P07 的衔接设计
  - P03 对聋人（P06）的适配：P03 的核心产品 MindRing 以语音为主要交互路径（"AI-powered voice journaling"），但 newLearning 的知识树提示词和辅导提示词本身是纯文字工具，可直接使用，绕开语音路径，对 P06 的使用无结构性障碍

### P07（张晨曦 / chenxizhang）

- **角色**: 知识深化工程脚手架提供者，同时将 P06 的学习过程纳入自身学术研究，形成耦合工作流

- **贡献**:
  - 提供论文工程化协作系统中的 /deep-read 命令（输出结构化笔记：核心论点 + 方法 + 与 P06 研究的关联性），可直接处理 P06 的周更学习笔记（1000-2000 字结构化文字，本质为 markdown 格式）
  - 提供分层文献索引机制（摘要→章节→段落，解决浅检索问题：问 AI 文献里有没有 X，它说没有但实际上有）
  - 提供 PR 可追溯系统（每次修改可追溯，解决"改了一段话忘了当时为什么这么写"的问题）
  - 提供 Mem0 向量记忆集成（AI 跨对话记住偏好、洞见、进度），解决 AI 记忆跨对话失效的已知卡点
  - P07 的六个项目（GlimpseMe、论文工程化、表里世界博客、文本存在论、Digital Desktop、Prompt Atelier）的信息流全部为文字驱动、本地存储、异步触发，跨项目一致，无任何实时语音依赖——对 P06（聋人）的协作摩擦结构性为零，不是绕开语音约束，而是根本不存在语音约束

- **收益**:
  - P06 的每周学习笔记（正在进行中的"零背景者借 AI 介入前沿领域"的跨界实践记录）对 P07 论文研究有纵向案例价值——P07 的论文研究 Z 世代员工通过 AI 素养获得感知优势并向上影响知识标准定义，P06 是这个命题的一个正在进行中的活体案例
  - P07 使用 /deep-read 处理 P06 的内容时，P06 同时获得结构化深读产出（两者的交换是嵌套的：P07 的工具以 P06 的内容为操作对象，P06 在贡献案例数据的同时获得工具输出），不是两项独立的单向给予
  - 条件：P06 须明确同意让自己的学习过程被 P07 作为研究材料使用（此条件在 formulation 中不可见，需要 P06 主动确认）

- **成本**:
  - 论文工程化系统有"AI 只执行部分工作流"的已知问题（P07 Profile 踩坑记录原文），在 P06 的生产过程中引入这套工具会在前期产生新的操作性摩擦，P06 需要承担学习曲线
  - P07 需要分出研究注意力持续跟踪 P06 的学习记录，以维持纵向案例观察

- **依赖**:
  - P07 能够帮助 P06"深化"的前提是 P03 的入域工具链已帮助 P06 建立了初步领域认知地图（时序依赖：P03 的工具先行，P07 的工具后续）
  - P06 须同意被研究，且须接受 git + markdown + PR 工作流的学习曲线

## 三、协作顺序与节点

**节点 A：P06 路径选择**

P06 须先确认自己的学习路径哲学偏好。P03 的工具链预设"先建立概念地图再行动"，P04 的路径预设"先锁定验证目标再在做中学"，两者在第一步上结构性互斥。P06 选定之后，后续工具的使用时序才能确定。

- 选择 P03 路径（先框架）：P03 的入域工具链（知识树→辅导→沙盒）在前，P07 的深化脚手架（/deep-read→PR→Mem0）在后，P03 负责设计接口让两者衔接
- 选择 P04 路径（先目标）：P04 的路径案例作为参照，P06 先锁定一个具体子领域的项目目标，P07 的工具在项目进行中接入，P03 的知识树工具降为按需使用

**节点 B（P03 路径下）：入域阶段**

P06 使用 P03 的知识树提示词建立四个目标领域（或优先选定一个领域）的核心概念地图 → P03 的 autogen 沙盒演示帮助 P06 在模拟场景中练习领域场景理解 → P03 将入域工具链产出（markdown 文件）输出到 P07 的系统入口，完成接口衔接。

**前提 B 能开始 C**：P03 完成接口约定设计并与 P07 确认 /deep-read 输入格式规范。

**节点 C：深化阶段（不依赖节点 B 的先行完成，若 P06 选 P04 路径则直接从这里开始）**

P06 使用 P07 的论文工程化系统（/deep-read 处理 P06 的周更笔记，PR 系统追踪修改，Mem0 维持跨对话记忆），在项目驱动中持续生产和积累工件。P07 同时对 P06 的学习过程做结构化研究记录（需要 P06 同意被研究）。

**前提 C 能持续**：P06 同意被 P07 作为研究材料，且接受学习曲线（git + markdown + PR 工作流的操作成本在前期为负）。

**节点 D：P04 研究引用（可并行于 A/B/C，独立通道）**

P04 向 P07 提供书籍信息（书名、作者、ISBN、贡献章节）。P07 将 P04 的出版记录作为论文一手文献引用，支持"非专家可以进入前沿研究"的前提命题论证。此通道与 A/B/C 完全独立，不存在时序依赖，不受 P06 路径选择的影响。

**格式障碍说明**：P04 的已有工件产出（竞赛代码、产品原型、答辩材料）与 P07 的工程脚手架输入要求（git PR 历史、可追溯 markdown 节点）不兼容，因此 P04↔P07 的工程合作通道（P04 工件被 P07 工具处理以帮助 D01）被封堵。但研究引用通道（P04 书籍被 P07 论文引用）不受影响，两个通道独立。

## 四、残余张力

**缺口一（首要，需新一轮场发现）：三个目标领域无人覆盖**

D01 的四个目标领域中，本次结晶的三位参与者叠加后只能覆盖"个人提升"（P03 的 MindRing/onething 方向，P04 的职场启航小助手）。"星球探索""长寿科技""AI 安全"三个领域，三位参与者均无实质内容积累，无任何直接工件记录。

三位参与者提供的是"如何生产知识"的工具链和路径支撑，但 D01 要在这三个领域内产出前沿知识工件，还需要在目标领域内有接触点——这个接触点在本次结晶中完全缺失。这是参与者覆盖缺口，不是本次结晶能消解的张力维度。

**建议动作**：以"星球探索""长寿科技""AI 安全"三个领域为筛选条件，重新触发场匹配，寻找在这三个领域有实质内容积累的知识生产者进入下一轮结晶。

**缺口二：P06 的交换势（E）价值边界未被外部验证**

P06 的交换势（可持续输出结构化学习笔记和概念框架）在 formulation 时即被标注为"部分充分"——有写作能力实证（博客），但工件被外部认定的记录为零（27 次阅读）。本次结晶涉及的两个条件互补（P07↔P06 的耦合工作流、P04↔P06 的条件互补）都以 P06 的输出质量达到一定水平为前提条件，该前提当前未满足。P03 所需的"内容伙伴"价值，也依赖 P06 的输出被外部验证后才能激活。

P07↔P06 的耦合工作流还需要 P06 明确同意让学习过程被作为研究材料，此同意条件在 formulation 中不可见，悬而未决。

**缺口三：P06 的阻碍函数 B 仍是空白**

formulation 标注 P06 的 B=[data_insufficient]，无具体失败案例、无记录过的认知断裂点、无资源缺口的具体描述。P04 的案例提供了路径存在性的参照，P07 的系统提供了四个已知卡点的解法，但 P06 实际会在哪个环节遇到阻碍，仍未知。工具和路径参照能覆盖的是"候选卡点 + 候选解法"，不是 P06 的真实阻碍。这个信息差只能通过 P06 实际开始执行后积累的数据来填补。
