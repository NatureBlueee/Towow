## 三、Prompt 问题诊断

### 问题 1：收敛失败（最严重）

**现象**：6 轮未收敛，催化 Agent 从未输出"本轮没有新发现"。

**根因分析**：
- 催化 Agent 在每轮都产出丰富的分析内容（~3.2K chars），不断深化而非收敛
- 催化 prompt 中的收敛自检（"指出你没有新发现时要明确说"）未被执行
- 催化 Agent 表现得更像**战略顾问**（持续产出洞察）而非**信息差消除者**（消除完毕则停止）

**v1 修复方向**：
1. 强制结构化输出：要求催化在输出末尾写 `收敛状态：本轮新发现 N 个` 或 `收敛状态：本轮没有新发现`
2. 区分"新发现"和"深化已知发现"：只有前者算新发现
3. 程序化收敛检测：计算连续轮次输出的语义相似度，超过阈值则触发收敛
4. 明确定义"新发现"：新的跨参与者关系 or 新的信息差，不包括对已有发现的分析深化

### 问题 2：催化约束违反（显著）

**现象**：催化 Agent 频繁推荐行动方案，违反"不推荐行动方案"约束。

**具体违反**：
- R1："催化建议"整段给每人具体建议（"你愿意在项目开始前跟 CEO 明确讨论…吗？"）
- R1："给 CEO 的问题清单"——这是方案不是观察
- R2：提出"元规则"（认知整合优先于投票）——这是行动建议
- R4：给出"瞬间分享协议"的详细操作流程
- R6：设计"描述对抗工作坊"的完整时间表

**影响**：催化在循环内做了方案生成器的工作，导致最终的 plan 难以区分哪些是催化推动的、哪些是端侧自发涌现的。

**v1 修复方向**：
1. 强化约束措辞：从"不推荐行动方案"到"你的每一句话都只能描述关系或指出信息差，绝不能包含'建议''应该''可以尝试'等行动性语言"
2. 添加反面示例
3. 可能需要 system prompt 级别的约束（而非 user prompt 中的指令）

### 问题 3：跨语义翻译能力不足（显著）

**现象**：催化擅长结构分析但不做跨语义翻译——没有"A 说的 X 和 B 说的 Y 可能在描述同一种能力"的输出。

**根因**：
- 催化 prompt 中翻译任务的描述偏抽象（"当你发现两个人可能在说同一件事时，指出来"）
- 没有给出具体的翻译输出格式示例
- 分析性输出（结构、拓扑、动力学）比翻译性输出更"显眼"，LLM 自然偏向前者

**v1 修复方向**：
1. 在催化 prompt 中增加翻译输出的具体示例
2. 要求催化每轮必须输出至少一条"跨语义翻译"（即使是"本轮未发现跨语义等价表达"）
3. 强调翻译优先级高于结构分析

### 问题 4：端侧同质化（中等）

**现象**：三个端侧 Agent 的回复长度一致（~2K chars），风格逐渐趋同，创造性张力减弱。

**根因**：
- 合成 Profile 带有相似的结构（都有"我擅长""我需要""我在想"）
- Sonnet 模型的"合作倾向"较强，端侧 Agent 倾向于同意催化观察
- 催化的详细建议压缩了端侧的自主表达空间

**v1 修复方向**：
1. Profile 结构差异化（有人写长文，有人写短句，有人用比喻）
2. 端侧 prompt 添加"你不需要回应主持人的所有观察，如果你不同意某个观察，说出来"
3. 考虑对不同端侧使用不同温度参数

## 四、方案生成器评估

### 优点
- **具体可执行**：5 天测试周 + 小时级时间表
- **四方完整**：每个参与者（含 CEO）都有角色/贡献/收益/成本/依赖
- **包含 GO/NO-GO 决策点**：不是一味推进
- **协作顺序清晰**：阶段 0 → 阶段 1（测试周）→ 后续

### 问题
- **受催化越界影响**：方案中大量内容（测试周流程、描述对抗工作坊、瞬间分享协议）实际上来自催化在循环内的推荐，而非端侧的自发涌现
- **过于精密**：一个 5 天测试周被设计到小时级别，可能过度工程化
- **成本假设**：80-120 万人民币的报价来自端侧 Agent 的估计，方案生成器直接采用，没有质疑

### 评级：3.5/5（好但受催化越界污染）

## 五、假说验证

| 假说 | 结果 | 说明 |
|------|------|------|
| H1: 五级优先级是否有效指导催化注意力 | **部分有效** | 催化确实关注了"未被发现的匹配"和"隐藏的阻碍"，但也大量产出不在五级框架内的结构分析 |
| H2: "可能"语气是否让催化保持克制 | **失败** | 催化频繁越界给出行动建议，"可能"语气约束力不足 |
| H3: 收敛自检是否可靠 | **失败** | 催化从未执行收敛自检，6 轮未收敛 |
| H4: 无结构化格式时催化输出是否清晰 | **成功** | 催化自发选择了清晰的分层结构（涌现模式/张力/动力学），可读性很好 |

## 六、对协议设计的影响

### 确认的假设
1. **多轮循环有真实价值**：每一轮都产出了前一轮不可能产出的内容（R2 的"学习型联盟"、R4 的"三层架构"、R6 的"协同即产品"）
2. **催化 Agent 确实能看到端侧看不到的东西**：权力博弈、时间冲突、串行依赖等
3. **端侧 Agent 能基于 Profile 产出有说服力的角色扮演**：即使是合成数据
4. **方案生成器作为独立步骤是正确的**：分离催化（观察）和方案生成（行动）在概念上是对的

### 需要修正的假设
1. **"协议止于可见性"需要更强的工程保障**：prompt 级别的约束不够，可能需要输出格式强制
2. **收敛机制需要从"自检"改为"程序化检测"**：LLM 的自检不可靠
3. **催化的核心能力需要从"结构分析"重新聚焦到"跨语义翻译"**：v0 prompt 没有足够强调翻译

### 下一步
1. **v1 催化 prompt**：强化约束 + 强调翻译 + 强制收敛格式
2. **v1 端侧 prompt**：增加异议表达引导
3. **程序化收敛检测**：实现基于语义相似度的自动收敛判断
4. **真实 Profile 测试**：等社区 Profile 收集后重跑
