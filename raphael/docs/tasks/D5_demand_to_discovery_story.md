# D5 — "从需求到发现" 案例故事

## 为什么做这件事

通爻网络的核心主张是"响应范式"——你发出信号，能帮你的人自己出现，而且你可能发现自己真正需要的跟最初以为的完全不一样。这个主张说起来一句话就够了，但"理解"和"感受到"之间有一道鸿沟。

概念层的文章我们已经有了（投影、谦逊、回声）。但概念再精妙，如果读者没有"体验过"那种认知转变——从确定到动摇到发现到惊喜——他就只是"知道"这个理念，而不是"相信"它。

这篇案例故事要做的事情是：**让读者在阅读中经历一次完整的认知转变。** 不是告诉他"通爻能做什么"，而是让他跟着主人公走一遍，走完之后自己意识到："原来我也有过类似的经历——以为需要 X，其实需要 Y，甚至还有一个 Z 是我从来没想过的。"

目标读者是所有人，尤其是非技术读者。这不是技术博客，是故事。

## 你要回答什么问题

1. **一个人以为自己需要什么？** 故事的起点——一个具体的、读者能共情的需求。这个需求要足够"合理"，让读者觉得"我可能也会这么想"。
2. **发生了什么让他动摇了？** 不是被说教了，而是在互动中自然地意识到原来的想法有盲点。关键：动摇来自他没想到的人/信息，不是来自权威。
3. **他真正需要的是什么？** 认知转变的核心——原来的需求不是"错了"，而是"不够深"。真正的需求在更底层。
4. **有没有一个"意外发现"？** 超出需求本身的、他从来不知道存在的东西。这是响应范式最独特的价值——搜索找不到你不知道要搜什么的东西。
5. **这个故事让读者想到了自己什么？** 文章结束后，读者脑子里浮现的应该是自己生活中的某个场景，而不是"通爻真厉害"。

### 场景要素（demo\_scenario.json 摘要）

**原始需求**：创业者想找一个技术合伙人（全栈，给 15% 股份，业余时间合作），做一个自由职业者管理工具。

**参与者**：
- 程序员 Alex（全栈，有兴趣但先问细节）
- 程序员小林（自由开发者，建议按小时咨询）
- 外包工作室（报价 5 万整包开发）
- Cursor（AI 编程工具，能生成 70% 代码）
- 产品教练老王（挑战核心假设："你验证过需求吗？"）
- Notion 模板作者（已有 200 用户的管理模板，带来意外发现："用户问得最多的是怎么找客户"）
- Bubble（No-code 平台，两周做出 MVP）

**认知转变弧线**：
1. 确定："我需要技术合伙人"
2. 动摇：产品教练问"你验证过需求吗？"，Notion 模板作者说"80% 功能已经有了"
3. 发现：真正需要的不是技术合伙人，而是"快速验证需求的能力"
4. 惊喜：Notion 模板用户最关心的是"怎么找客户"——这可能改变产品方向

**结果对比**：
- 原方案：15% 股份 + 几个月找人 + 需求未验证风险
- 协商后方案：约 1000 元 + 1-3 周验证 + 保留股份 + 可能发现更好的方向

### 已有结论（不需要解释的概念）

- **响应范式**的价值已在概念层被论述过（01\_投影.md、Arch Skill）
- **"需求 != 要求"** 是通爻的核心设计原则之一
- **"超额满足"** 和 **"发现未知"** 是响应范式的两个核心优势

## 子任务分解

### D5.1: 选择/设计案例场景（0.5 天）

从 demo\_scenario.json 出发，决定：

**选项 A：直接用"找技术合伙人"场景**
- 优势：素材完整，协商过程已设计好，认知转变弧线清晰
- 风险：可能太"设计感"，读者觉得是虚构的

**选项 B：基于"找技术合伙人"改编，加入更多生活细节**
- 优势：保留骨架但更真实
- 做法：给主人公一个名字、一个具体的生活情境、一些情绪变化

**选项 C：自创一个新场景**
- 优势：完全自由，可以选最能打动人的场景
- 风险：需要从头设计认知转变弧线
- 可选场景方向：找工作、装修房子、选学校、创业找投资...

输出：选定场景 + 场景要素清单（主人公、参与者、认知转变弧线、意外发现）。

### D5.2: 设计叙事结构（0.5 天）

设计认知转变的叙事弧线。核心节奏：

```
1. 起点：确定的需求（"我需要 X"）
   - 读者要觉得"这个需求很合理，我也会这么想"

2. 展开：响应开始出现
   - 有直接回应 X 的人
   - 有问"你为什么需要 X"的人
   - 有提供意想不到东西的人

3. 动摇：意识到原来的假设有盲点
   - 不是被人说教了
   - 而是信息本身让他自己意识到的

4. 发现：真正的需求浮现
   - "我以为需要技术合伙人，其实需要的是验证能力"
   - 新方案比原方案更好（不只是更便宜，是更对）

5. 惊喜：完全没想到的东西
   - "用户问得最多的不是项目管理，而是怎么找客户"
   - 这个信息是搜索永远找不到的——因为你不知道要搜什么

6. 回味：故事结束后的余韵
   - 主人公没有"正确答案"，但有了"更好的问题"
   - 读者想到自己生活中类似的经历
```

输出：叙事大纲，标注每个节奏点的情绪变化。

### D5.3: 撰写故事初稿（2-3 天）

1500-3000 字，核心要求：

- **有主人公**：有名字、有处境、有情绪。不是"某创业者"，是一个读者能想象的具体的人
- **有对话**：不需要每一句都写，但关键的认知转折点要有直接引语
- **有情绪变化**：从自信 -> 犹豫 -> 恍然 -> 兴奋 -> 平静，读者要能感受到这个变化
- **不解释机制**：不说"系统通过共振检测找到了相关响应者"，只说"小王发了一条消息，然后收到了几个完全没想到的回复"
- **结尾开放**：故事不以"主人公成功了"结尾，而以"主人公有了新的出发点"结尾

视角选择：第一人称（更沉浸）或第三人称（更客观），由作者决定。

### D5.4: 嵌入概念（1 天）

在故事写完后，回顾检查：通爻的核心概念是否自然地体现在故事中？

| 概念 | 在故事中如何体现 | 注意 |
|------|-----------------|------|
| 投影 | 不同人从不同角度看同一个需求 | 不用"投影"这个词 |
| 共振 | 有些完全没想到的人会主动回应 | 不用"共振"这个词 |
| 协商 | 不是一次匹配，是多轮对话中方案逐渐成型 | 不用"协商"这个词，用生活化的描述 |
| 需求 != 要求 | 主人公以为需要 X，发现真正需要 Y | 让读者自己体会，不要点破 |
| 超额满足 | 意外发现 Z（从未想过的信息） | 这是故事的高光时刻 |
| 响应范式 | 帮助来自主人公没想到会来的方向 | 不用"响应范式"，只展示效果 |

关键原则：**概念要像水一样渗透在故事里，而不是像标签一样贴在故事上。** 如果一个概念必须用术语才能表达，说明故事还没写到位。

### D5.5: 内部审阅 + 修改（1 天）

审阅清单：
- [ ] **共情测试**：读完前 200 字，读者能不能说"这个人的处境我理解"？
- [ ] **认知转变测试**：读到"动摇"时，读者是不是也跟着动摇了？还是觉得"这也太刻意了"？
- [ ] **意外感测试**："意外发现"是不是真的让人意外？还是读者早就猜到了？
- [ ] **说教检测**：全文有没有任何一处在"告诉读者应该怎么想"？如果有，删掉或改写
- [ ] **术语检测**：全文有没有出现"投影"、"共振"、"协商范式"、"响应范式"等通爻术语？如果有，替换为日常语言
- [ ] **通爻检测**：文章主体中是否提到通爻？主体中不应该出现品牌名。末尾可以简短提及
- [ ] **余韵测试**：读完最后一段，读者脑子里想的是自己的什么经历？

## 做完了是什么样

### 产出格式
- Markdown 文件
- 纯叙事，无代码、无架构图、无技术术语
- 可以有少量引语/对话

### 产出规模
- 1500-3000 字（中文）
- 约 3-5 分钟阅读时间

### 质量标准

**"做完了"**：
- 有完整的认知转变弧线（确定 -> 动摇 -> 发现 -> 惊喜）
- 有具体的主人公和情境（不是抽象的"某用户"）
- 没有通爻术语（投影、共振、协商等）
- 通爻的核心概念自然地体现在故事中
- 文章可以独立阅读，不需要任何关于通爻的前置知识

**"做得好"**：
- 读者读完后会想到自己生活中"以为需要 X，其实需要 Y"的经历
- 意外发现环节让读者真的觉得意外（不是猜到的）
- 主人公的情绪变化让人共情，不觉得刻意
- 结尾让人安静下来，而不是兴奋起来——好的故事结尾是"嗯"，不是"哇"
- 非技术读者和技术读者都能享受这个故事

## 你必须遵守的

1. **不用通爻术语**：文章主体中不能出现"投影"、"共振"、"协商"、"响应范式"、"Agent"等通爻/技术术语。这是最硬的约束。末尾可以用一两句提及通爻和背景。
2. **不传教**：故事不是为了证明"通爻多好"。故事是为了让读者经历一种体验——如果这个体验本身有价值，读者自己会去了解通爻。
3. **不给标准答案**：故事的结尾不是"主人公找到了正确答案"。是"主人公带着更好的问题出发了"。生活不给标准答案，故事也不应该。
4. **主人公要真实**：不是完美的创业者，是一个有焦虑、有盲点、会犯错的普通人。读者要能在主人公身上看到自己。
5. **风格对齐**：参考已有三篇文章的共同底色——有生活感、有自省、不学术、结尾有余韵。这篇更偏叙事，但底色一致。
6. **场景核心不能偏**：无论选用还是改编场景，"认知转变"（以为需要 X，发现需要 Y，意外得到 Z）这条主线不能丢。这是整篇文章存在的理由。

## 你可以自己决定的

1. **场景选择**：可以直接用"找技术合伙人"（demo\_scenario.json），可以改编，也可以完全自创新场景。只要能承载认知转变的弧线即可
2. **叙事视角**：第一人称（更沉浸、更私密）或第三人称（更客观、更有距离感）
3. **主人公设定**：性别、年龄、职业、性格——完全自由，选最能让目标读者共情的
4. **对话密度**：可以大量使用对话（像小说），也可以少用对话（像散文）
5. **意外发现的具体内容**：可以用 demo\_scenario.json 中的"用户问得最多的是怎么找客户"，也可以设计一个更出人意料的发现
6. **结尾的形式**：可以是主人公的独白、一个开放性问题、一个生活细节——任何能产生余韵的形式
7. **标题**：可以用"从需求到发现"，也可以自拟。好的标题会让人好奇但不剧透
