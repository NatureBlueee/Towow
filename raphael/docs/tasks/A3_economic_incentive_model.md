> 创建日期：2026-02-07
> 任务类型：经济设计 × 方向探索
> 优先级：Tier 2（#5）
> PRD 状态：已细化
> 关联任务：A6（Sui 链成本实测，理解链上操作的经济约束）

---

## 这个任务在项目中的位置

通爻（ToWow）是一个 AI Agent 协作网络。用户发出需求信号，网络中相关的 Agent 自主响应、协商、共创方案。

但一个根本问题是：**为什么有人愿意让自己的 Agent 参与？** 如果没有合理的激励机制，Agent 数量不会增长，网络就无法形成。通爻需要回答"参与者为什么来、为什么留、如何获得价值"。

**A3 不是设计通爻的经济系统**（那是后续工作），而是**方向探索**——梳理学术界和实践中的激励模型，分析每种模型在通爻"响应范式"下的适用性，提出候选方向。这份报告无论通爻未来选择哪种激励机制，都是有价值的参考。

```
[A3: 经济激励探索（你在这里）] → [核心团队决策] → [V1/V2 激励机制实现]
                                                        ↓
                                              [A6: Sui 链成本数据]
```

---

## 为什么做这件事

通爻网络要运转起来，需要人（和他们的 Agent）持续参与：发布需求、响应需求、参与协商、执行承诺、提供反馈。但"为什么有人愿意参与？"这个问题目前没有系统性的答案。

架构设计已经明确了**回声信号**的技术机制（见 Section 11：链上执行数据回流更新 Agent Profile），也初步涉及了支付问题（见 Section 11.4.4：V1 用信用额度/模拟货币，未来探索方向包括积分/信誉系统），但**经济激励层面的整体设计**是缺失的：

- 谁为谁的什么行为付费（或给予奖励）？
- 什么行为应该被激励？什么行为应该被惩罚？
- "高质量响应"和"低质量响应"如何区分？由谁区分？
- 如何避免"劣币驱逐良币"（低质量 Agent 充斥网络）？
- 如何处理冷启动（新 Agent 没有声誉，但可能很优秀）？

这不是一个设计任务（我们不需要你设计通爻的经济系统），而是一个**方向探索任务**：梳理可选的激励模型，分析每种模型在通爻"响应范式"下的适用性，提出候选方向供后续深入。

**为什么现在值得做**：经济学理论和博弈论模型不会因为我们的架构变化而作废。这份探索报告无论通爻未来选择哪种激励机制，都是有价值的参考。架构韧性高。

## 你要回答什么问题

**核心问题**：在一个"响应范式"的网络中（不是搜索/交易平台，而是信号广播 + 自主响应 + 协商共创），应该用什么样的经济激励机制来吸引和留住高质量的参与者？

具体来说：

1. **价值流梳理**：通爻网络中的价值是如何产生、流动和消耗的？谁是价值的创造者？谁是价值的消费者？有哪些"价值事件"（value events）是可以被锚定和量化的？
2. **现有模型调研**：学术界和实践中有哪些主要的激励模型（Token 经济、声誉系统、贡献积分、质押机制、拍卖、注意力经济、共享收益等）？每种模型的核心假设和适用条件是什么？
3. **响应范式的特殊性**：通爻的"响应范式"与传统搜索/交易平台有什么本质区别？这些区别如何影响激励模型的选择？（例如：搜索平台的"点击付费"模型是否适用？为什么？）
4. **候选方案**：结合 1-3 的分析，提出 2-3 个候选激励方案，每个方案包含核心机制、适用条件、trade-off、风险。
5. **冷启动与公平性**：任何激励模型都面临"马太效应"（强者愈强）问题。在通爻的场景下，如何设计让新参与者有公平的机会？

## 我们提供什么

### 概念着陆（通爻术语 → A3 语境中的含义）

| 通爻术语 | 在 A3 中意味着 |
|---------|---------------|
| 响应范式 | 通爻的核心范式——不是"搜索匹配"，而是"广播信号 → Agent 自主判断相关性 → 主动响应"。这意味着传统平台的"点击付费""竞价排名"等激励模型可能不适用 |
| 共振（Resonance） | Agent 判断"这个需求跟我有关"的过程。共振是 Agent 的主动行为——它消耗计算资源来检测相关性。如何激励 Agent 持续做这件事？ |
| 回声（Echo） | 协作执行后，链上产生的真实反馈信号。这是通爻区别于传统评价系统的关键——质量信号来自链上执行数据，不是人工评分 |
| 投影即函数 | Agent 不是"发布了个人简历"，而是"每次被需求触发时，从数据源重新投影一次"。这意味着 Agent 的"参与成本"和传统平台的"发布成本"不同 |
| 场景（Scene） | 协作发生的有界上下文。场景组织者设定参数（如期望响应数 k*）。激励模型需要考虑场景组织者的角色和激励 |
| k* 机制 | 期望响应数量——一个业务参数，而非固定阈值。激励模型需要与 k* 兼容（不是所有 Agent 都应该被激励响应同一个需求） |

### 设计原则（与 A3 的关系）

| # | 原则 | 与 A3 的关系 |
|---|------|-------------|
| 0.6 | 需求 ≠ 要求 | 需求是抽象的张力，要求是假定的解决方案。激励模型不能假设"用户知道自己要什么"，需要激励"帮助用户发现真实需求"的行为 |
| 0.5 | 代码保障 > Prompt 保障 | 激励和惩罚的执行应该由代码（如链上合约）保障，不能依赖"Prompt 告诉 Agent 要诚实" |
| 0.8 | 投影是基本操作 | 每次投影都消耗计算资源——激励模型需要考虑这个"参与成本" |

### 通爻网络中的关键角色和行为

做激励分析时，这些是你需要考虑的角色和行为：

| 角色 | 关键行为 | 行为的价值 |
|------|---------|-----------|
| **需求发布者** | 发布需求、确认方案、执行承诺、提供反馈 | 为网络带来"待满足的需求"（驱动力） |
| **响应者（Agent）** | 检测共振、生成 Offer、参与协商、执行承诺 | 为需求提供解决方案（供给侧） |
| **Center Agent** | 聚合 Offer、识别缺口、生成方案 | 协调多方，创造 1+1>2 的组合价值 |
| **场景组织者** | 创建场景、设定参数（k*）、管理准入 | 定义协作的上下文（环境营造） |

### 通爻与传统平台的关键差异

在评估激励模型时，请特别注意以下差异——这些使得很多传统模型不能直接套用：

| 维度 | 传统搜索/交易平台 | 通爻网络 |
|------|-----------------|---------|
| **匹配方式** | 平台中心化匹配 | Agent 端侧自主判断是否共振 |
| **价值发现** | 已知需求找已知供给 | 可能发现"你不知道你需要"的价值 |
| **交易结构** | 一对一交易 | 多方协商、方案组合 |
| **质量信号** | 评分/评价（人工） | 链上执行数据（回声，不可伪造） |
| **供给侧成本** | 发布服务/商品 | 运行 Agent + 计算共振 + 生成 Offer（持续的计算成本） |
| **核心壁垒** | 数据/流量 | 网络效应（Agent 越多，需求越能被满足，越多人用） |

### WOWOK 区块链基础设施

通爻的执行阶段基于 WOWOK（Sui 区块链协议），提供以下链上能力，是激励模型可以利用的基础设施：

- **Treasury**：链上支付管理
- **Guard**：条件验证引擎（可以定义任意条件）
- **Progress**：workflow 执行跟踪
- **Order**：交易记录
- **Service**：服务门户（含承诺、价格、workflow）

详见 `docs/ARCHITECTURE_DESIGN.md` Section 11.2。

## 子任务分解

### A3.1 通爻网络价值流梳理

**描述**：在通爻网络的完整流程（需求发布 -> formulation -> 广播 -> 共振 -> Offer 生成 -> 协商 -> 方案 -> 执行 -> 回声）中，梳理"价值在哪里产生、如何流动、被谁消耗"。识别所有可锚定的"价值事件"。

关键问题：
- 需求发布本身是否有价值？对谁有价值？
- "高质量 Offer"的价值由谁评判？（Center 聚合时？执行后的回声？）
- "意外发现"（用户得到了自己没想到的好方案）的价值如何归属？
- 协商过程中的"信息贡献"如何量化？

**依赖**：无（需要先理解通爻的流程，可从架构文档入手）

**交付物**：价值流图（可以是文字描述或图示），标注每个环节的价值事件和利益相关者

### A3.2 现有激励模型调研

**描述**：系统调研学术界和实践中的主要激励模型。建议覆盖（但不限于）：

- **Token 经济**：加密货币/平台代币激励（如 Filecoin、Helium、BAT）
- **声誉系统**：基于历史行为的信誉评分（如 eBay 评价、Stack Overflow 积分）
- **贡献积分/信用系统**：非货币化的贡献记录（如开源社区贡献者等级）
- **质押机制**：参与者需要质押资产来保证行为质量（如 PoS、prediction market）
- **拍卖机制**：竞价获取资源/曝光（如 Google Ads、频谱拍卖）
- **注意力经济**：基于注意力分配的价值分配（如 Brave/BAT）
- **共享收益**：平台收入按贡献比例分配（如 YouTube 分成、合作社模式）
- **博弈论机制设计**：VCG 拍卖、revelation principle、机制设计理论

每种模型需要分析：核心假设、适用条件、已知问题、代表性实践案例。

**依赖**：无（可与 A3.1 并行）

**交付物**：激励模型分类综述（每类 300-500 字），含核心机制图示和代表案例

### A3.3 各模型在响应范式下的适用性分析

**描述**：这是本探索最有价值的部分。将 A3.2 中的模型放到通爻的"响应范式"场景下评估。

评估维度建议：

| 维度 | 说明 |
|------|------|
| **与响应范式的兼容性** | 这个模型能否适配"信号广播 + 端侧自主响应"的范式？还是假设了中心化的匹配/分配？ |
| **意外价值的激励** | 这个模型能否激励 Agent 去发现和提供"用户自己没想到"的价值？还是只激励精确匹配？ |
| **质量信号来源** | 这个模型依赖什么信号来判断"好"和"坏"？这些信号在通爻中能否获得？ |
| **冷启动友好度** | 新参与者在这个模型下能否获得公平的机会？ |
| **抗博弈性** | 参与者能否通过策略性行为（刷量、串通、女巫攻击）来获取不当利益？ |
| **实现复杂度** | 基于 WOWOK 链上基础设施，这个模型实现起来有多难？ |
| **监管兼容性** | 在中国大陆的监管环境下，这个模型是否可行？（特别是涉及 Token 的模型） |

**依赖**：A3.1 + A3.2

**交付物**：适用性分析表（模型 x 维度），含每个单元格的详细讨论

### A3.4 候选方案提出与 trade-off 分析

**描述**：基于 A3.1-A3.3 的分析，提出 2-3 个适合通爻网络的候选激励方案。每个方案应该是一个**组合**（可能融合多种模型），不是某单一模型的直接套用。

每个候选方案需包含：
- 核心机制描述（一段话说清楚）
- 与通爻流程的对接点（在哪些"价值事件"上触发激励/惩罚？）
- 最大的优势（为什么选这个？）
- 最大的风险（什么情况下会失败？）
- V1 可行性（能否用最简单的方式在 V1 中验证？）
- 与 WOWOK 基础设施的对接方式

**依赖**：A3.3

**交付物**：2-3 个候选方案文档（每个 800-1500 字），含 trade-off 对比表

### A3.5 撰写方向探索报告

**描述**：综合以上内容，撰写完整的方向探索报告。报告的定位是"为后续决策提供信息输入"，而非"给出最终答案"。报告末尾应明确标注：哪些是已有的学术共识，哪些是你的判断，哪些是需要进一步验证的假设。

**依赖**：A3.1 - A3.4

**交付物**：方向探索报告（6000-12000 字），结构为：引言、通爻价值流分析、激励模型综述、适用性分析、候选方案、结论与建议、参考文献

### 质量标准

**"做完了"的标准**：
- 完成了通爻价值流的梳理（至少识别 5 个以上"价值事件"）
- 调研了至少 5 种不同的激励模型
- 每种模型有清晰的核心机制描述和适用条件分析
- 适用性分析覆盖了所有主要评估维度
- 给出了至少 2 个候选方案并说明 trade-off

**"做得好"的标准**（超出基本要求）：
- 发现了"响应范式"与现有激励模型之间的根本性矛盾（如果有的话）
- 候选方案有具体的 V1 最小验证实验建议
- 分析了"激励失败"的场景和应对策略（反脆弱思维）
- 引入了跨领域的激励设计洞察（生物学、社会学、行为经济学等）
- 对通爻当前的 V1 策略（信用额度/模拟货币）给出了评价和改进建议

**"做得出色"的标准**（产生额外价值）：
- 提出了一种专门为"响应范式"设计的新型激励框架（而非套用现有模型）
- 发现了"意外价值"（用户发现了自己没想到的好方案）如何被激励——这是通爻最独特的场景
- 结合了 WOWOK 链上基础设施的具体能力，给出了可直接实现的激励合约设计草案
- 用博弈论分析了通爻场景下的纳什均衡和激励兼容性

## 你必须遵守的

1. **方向探索，不是最终设计**：你的任务是"列出可选方向并分析 trade-off"，不是"设计通爻的经济系统"。最终选择由核心团队做。
2. **区分理论和实证**：经济学模型在理论上成立不等于实践中可行。有实际案例的模型应明确标注案例；纯理论模型应标注"未经实践验证"。
3. **考虑中国大陆监管环境**：涉及 Token、加密货币的模型需要特别讨论监管风险。不能假设可以自由发行 Token。
4. **理解"响应范式"的特殊性**：通爻不是交易平台、不是推荐系统、不是劳务市场。激励模型必须与"信号广播 + 自主响应 + 协商共创"的范式兼容。不要简单套用现有平台的激励模型。
5. **诚实标注不确定性**：如果某个方向你认为有潜力但不确定，请明确说"这是我的判断，需要进一步验证"，而不是表述为确定的结论。

## 你可以自己决定的

1. **调研深度分配**：如果某个模型明显不适用于通爻，简要说明原因即可，不必深入。如果某个方向特别有潜力，可以花更多篇幅。
2. **模型覆盖范围**：建议的模型类别仅供参考。如果你知道其他相关的激励模型（如 DAO 治理、平台合作主义、互助网络等），欢迎加入。
3. **分析框架**：评估维度建议仅供参考，可以根据你的判断调整或增加。
4. **跨领域借鉴**：非常鼓励从其他领域借鉴激励设计思路（生物学的共生激励、社会学的互惠规范、行为经济学的nudge等）。
5. **候选方案数量**：2-3 个是参考，如果你认为有 4 个同样有价值的方向，可以都列出。
6. **写作语言**：中文或英文均可，优先中文。专业术语保留英文。
