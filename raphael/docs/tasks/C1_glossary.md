## 这个任务在项目中的位置

通爻（ToWow）的概念体系融合了三个领域：技术（HDC、超向量、SimHash）、哲学（投影、完备性、一自多我）和原创术语（共振、回声、透镜）。当与国际开发者、投资人、学术合作者沟通时，准确的中英翻译是一切的基础。

**错误的翻译会从根本上改变别人对通爻的理解。** 比如把"投影"翻译成"profile"会丢失"动态函数调用"的含义，让人误以为和 LinkedIn Profile 一样是静态存储；把"共振"翻译成"matching"会丢失"端侧自主判断"的含义，让人误以为是中心化匹配。

C1 就是建立这个翻译基础——一旦建立，就是所有英文内容的基石。

```
[C1: 术语表（你在这里）] → [所有英文文档和翻译]
                          → [国际沟通（投资人、开发者、学术）]
                          → [D1-D5 文章的英文版]
```

## 为什么做这件事

通爻的概念融合了技术（HDC、超向量）、哲学（投影、完备性）和原创术语（共振、回声）。和国际开发者/投资人沟通时，准确的中英翻译是基础。

错误的翻译会导致误解——比如把"投影"翻译成 "profile" 会丢失"动态函数调用"的含义；把"共振"翻译成 "matching" 会丢失"端侧自主判断"的含义。

术语表一旦建立，就是所有后续英文内容的基石。

## 你要回答什么问题

**核心问题**：如何建立中英双语术语表，保留每个术语的技术含义和哲学内涵？

**子问题**：
- 通爻的核心术语有哪些？（约 30-40 个）
- 每个术语的"不可丢失含义"是什么？（翻译时绝对不能丢的部分）
- 哪些术语有现成的英文对应？哪些需要原创翻译？
- 术语之间的层次关系是什么？（世界观级 / 机制级 / 角色级 / 流程级 / 技术级）
- 难点术语（如"自"、"投影"、"完备性"）有哪些候选译法？各自优劣？

## 子任务分解

### C1.1 — 从文档提取完整术语清单
**描述**：通读架构文档和 Design Log，提取所有核心术语（约 30-40 个）。记录每个术语的中文定义和首次出现位置。
**依赖**：无
**交付**：术语清单（中文名 + 定义 + 出处）

### C1.2 — 分层分类
**描述**：将术语按层次分类：世界观（投影、完备性、一自多我）、机制（共振、协商、回声）、角色（Edge Agent、Center Agent、Service Agent）、流程（formulation、offer、plan）、技术（HDC、SimHash、bundle）。
**依赖**：C1.1
**交付**：分层术语表

### C1.3 — 翻译与讨论
**描述**：为每个术语提出 2-3 个候选英文译法。分析优劣（哪个保留了含义？哪个更容易被技术人理解？哪个更容易被非技术人理解？）。对难点术语写详细讨论。
**依赖**：C1.2
**交付**：候选译法表 + 难点术语讨论（每个 200-300 字）

### C1.4 — 编写使用指南
**描述**：什么场景用什么翻译？（技术文档 vs 商务沟通 vs 学术论文）有没有绝对不能混用的？
**依赖**：C1.3
**交付**：使用指南（500-1000 字）

### C1.5 — 内部审阅 + 定稿
**描述**：提交给核心团队审阅。根据反馈修改。最终定稿。
**依赖**：C1.4
**交付**：定稿版术语表（`docs/GLOSSARY.md`）

## 做完了是什么样

### 产出清单

1. **术语对照表**（分层：世界观/机制/角色/流程/技术）
2. **难点术语讨论**（候选译法 + 优劣分析，至少 5 个难点术语）
3. **使用指南**（什么场景用什么翻译）

### 三级质量标准

**做完了（基本合格）**：
- 不是机翻——每个翻译都有理由
- 技术术语采用领域标准（如 HDC 就是 HDC）
- 哲学术语需保留内涵（不能翻成"听起来差不多但含义不同"的词）
- 覆盖至少 30 个核心术语

**做得好（超出预期）**：
- 难点术语的讨论有深度（参考了类似项目的翻译选择，如 Web3、AI Agent 领域的术语惯例）
- 使用指南区分了技术文档/商务沟通/学术论文三种场景
- 术语之间的层次关系和引用关系清晰可视化

**做得出色（产生额外价值）**：
- 创造了一个新的英文术语（如将"共振"翻译为一个新造词）并有说服力的论证
- 术语表可直接被自动化工具使用（如 i18n 翻译键、文档生成器）
- 包含了术语的"误用警告"——指出常见但错误的翻译（如 Projection ≠ Profile）及其会导致的误解

## 你必须遵守的

- 不是机翻
- 技术术语采用领域标准
- 哲学术语需保留内涵
- 不能为了"好听"牺牲准确性

## 你可以自己决定的

- 术语数量（可以多于或少于 30-40 个，但要覆盖核心）
- 翻译风格（学术风 vs 工程风 vs 通俗风）
- 难点术语的最终选择（但要说明理由）
