## 这个任务在项目中的位置

通爻（ToWow）是一个 AI Agent 协作网络，提出了"响应范式"来替代传统的"搜索范式"。但市面上已经有 AutoGen、CrewAI 等多 Agent 框架，有 LinkedIn、Boss 直聘等匹配平台，有 Fetch.ai 等去中心化 Agent 网络。

**A5 的核心使命不是"证明通爻更好"，而是精准识别：通爻的独特价值到底在哪？** 可能的发现包括："别人已经在做 X 了，但没人做 Y"——这个 Y 就是通爻的差异化定位。也可能发现"原来某个系统已经在做类似的事"——这同样有价值，能帮助通爻聚焦到真正独特的部分。

## 为什么做这件事

通爻提出了"响应范式"来替代"搜索范式"。外界会问："这和 AutoGen 有什么区别？和推荐算法有什么区别？"——如果我们自己说不清楚，别人更说不清楚。

这个分析不仅是为了"证明独特"，更是为了**精准识别独特价值在哪**。可能的发现包括："别人已经在做 X 了，但没人做 Y"——这就是通爻的差异化定位。

## 你要回答什么问题

**核心问题**：通爻网络和以下系统/框架的**范式级别**差异是什么？

**需要对标的系统**（三类）：
1. **多 Agent 框架**：AutoGen、CrewAI、MetaGPT、LangGraph、A2A Protocol
2. **匹配/推荐系统**：协同过滤、LinkedIn、Upwork、Boss 直聘
3. **去中心化/新范式**：Fetch.ai、SingularityNET、SecondMe

**子问题**：
- 每个系统的核心假设是什么？（它预设了什么？）
- 哪些能处理"未知需求"？（搜索范式 vs 响应范式的分水岭）
- 通爻的"投影"和其他系统的"画像/Profile"有何本质不同？（投影是函数调用，画像是静态存储）
- 是否有系统已经在做通爻想做的事？（如果有，通爻的增量在哪？）
- 各系统的 context 来源是什么？（单一 LLM vs 分布式 Agent 各自的 Profile）

## 我们提供什么

### 概念着陆（通爻术语 → A5 对比分析中的含义）

| 通爻术语 | 在对标分析中意味着 | 对标系统中的等价概念 |
|---------|-------------------|-------------------|
| 投影（Projection） | Agent 的能力表示是函数调用的结果，每次调用可能不同（因为数据源在更新） | vs AutoGen: 固定 system prompt; vs LinkedIn: 静态简历 |
| 共振（Resonance） | Agent 在端侧自主判断"这个需求跟我有没有关系"，不需要中心化匹配 | vs 推荐算法: 平台中心计算后分发; vs Fetch.ai: Agent 注册服务后被搜索 |
| 协商（Negotiation） | 多个 Agent 的 Offer 被聚合，产生一个新的整体方案——结果不是"排行榜"而是"共创" | vs CrewAI: 任务分配; vs Upwork: 竞标排序 |
| 响应范式 | "我不搜索，我发信号等响应"——核心差异化点 | vs 搜索范式: 用户主动输入条件过滤结果 |
| 认知转变 | 系统帮助用户发现"我以为需要 X，其实需要 Y" | 几乎没有对标系统专门设计这个能力 |
| 投影即函数 | Agent 是无状态的投影函数结果，不是有状态对象 | vs 所有对标系统: Agent/Profile 都是有状态实体 |

### 设计原则（与 A5 的关系）

| # | 原则 | 在对标分析中的意义 |
|---|------|-------------------|
| 0.8 | 投影是基本操作 | 这是通爻与所有对标系统的根本差异——对标系统都假设 Agent/Profile 是"实体"，通爻假设是"函数" |
| 0.6 | 需求 ≠ 要求 | 大多数对标系统假设用户知道自己要什么（搜索条件/任务描述）。通爻认为需求是需要被发现的 |
| 0.5 | 代码保障 > Prompt 保障 | 对标系统（如 AutoGen、CrewAI）主要靠 prompt 协调 Agent 行为；通爻靠代码逻辑保障 |

## 子任务分解

### A5.1 — 收集对标系统信息
**描述**：为三类对标系统收集核心信息——官方文档、白皮书、架构图、核心卖点。
**依赖**：无
**交付**：每个系统的 1 页信息摘要

### A5.2 — 分析各系统的核心假设
**描述**：提取每个系统的隐含假设。比如：AutoGen 假设"任务可以分解为步骤"；LinkedIn 假设"用户知道自己要找什么"。
**依赖**：A5.1
**交付**：假设对比表

### A5.3 — 制作多维度对比矩阵
**描述**：从范式层面（不是功能层面）对比。维度包括：context 来源、是否能处理未知需求、计算架构（中心化/分布式/端侧）、核心操作（搜索/投影/匹配/协商）。
**依赖**：A5.2
**交付**：对比矩阵表格

### A5.4 — 撰写差异分析
**描述**：基于对比矩阵，深入分析通爻的独特价值。要公平——承认别人做得好的，指出通爻的真正差异化点。
**依赖**：A5.3
**交付**：差异分析文章（5000-8000 字）

### A5.5 — 提炼定位一页纸
**描述**："通爻是什么，不是什么"——500 字以内，任何人 1 分钟读完就知道通爻和别人有什么不同。
**依赖**：A5.4
**交付**：定位一页纸（500 字）

## 做完了是什么样

### 产出清单

1. **对标分析报告**（5000-8000 字）：含对比表和差异分析
2. **一页纸定位**（500 字）："通爻是什么，不是什么"

### 三级质量标准

**做完了（基本合格）**：
- 至少覆盖 8 个对标系统（三类各至少 2 个）
- 范式级对比，不是功能列表
- 有引用和出处
- 公平对待各系统（不贬低别人来抬高自己）

**做得好（超出预期）**：
- 发现了通爻的"盲点"——别人已经解决而通爻还没考虑的问题
- 每个对标系统的核心假设被清晰提取（而非只描述功能）
- 对比矩阵有 insight，不只是信息罗列
- "一页纸定位"能被非技术人员理解

**做得出色（产生额外价值）**：
- 发现了某个对标系统的某个设计可以被通爻借鉴（具体说明怎么借鉴）
- 分析了通爻在各细分场景下的竞争优劣势，给出了"先打哪个场景"的战略建议
- 写出了可直接用于对外沟通的差异化叙事（投资人/合作伙伴/开发者各一版）
- 包含了 A2A Protocol（Google 2025）等最新进展的深度分析
