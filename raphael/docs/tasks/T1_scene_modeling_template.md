## 这个模板在项目中的位置

通爻（ToWow）是一个 AI Agent 协作网络，它的核心理念是"响应范式"——不是你搜索和筛选，而是你发出一个信号（"我需要帮助"），相关的人和资源自行判断"这跟我有没有关系"，然后主动响应。

通爻相信这种范式在很多领域都有价值——招聘、餐饮推荐、技能交换、医疗咨询……但"相信"不等于"验证"。**每个领域有自己独特的用户行为、决策逻辑和痛点。** 直接套用通爻的技术框架，不先理解用户，做出来的东西就是"技术上能跑但没人用"。

**T1 就是做这个"理解用户"的工作。** 它是一个模板——当有来自特定领域的共建者时，用 T1 为他/她生成一个具体的场景建模任务。建模的产出不是代码，而是**对用户行为和需求的深入理解**。

```
[T1: 场景建模模板] ─实例化→ [B1: 黑客松组队场景]
                  ─实例化→ [B2: 招聘场景]
                  ─实例化→ [B3: 餐饮推荐]
                  ─实例化→ [B4: 技能交换]
                  ─实例化→ [B5: 更多领域...]
                                ↓
                     [T2: 独立小应用模板] → [具体应用]
```

> 这不是一个具体任务，而是**任务模板**。当有来自特定领域的共建者时，用这个模板为他/她生成具体任务。

## 这个模板适合谁

来自任何领域的人——招聘、餐饮、教育、医疗、游戏、法律、金融……只要他/她对自己的领域有深入理解。

不需要技术背景。需要的是：**对自己领域中"找人/找服务/找方案"这件事有切身体验。**

**用日常语言说**：如果你曾经在某个领域里帮别人介绍过人、推荐过方案、或者发现"他以为需要 X，其实需要 Y"——那你就有做这件事的经验和直觉。

## 怎么用这个模板

### 第一步：填写场景信息

```
场景名称：[你熟悉的领域]
核心问题：在这个领域，有没有"以为需要 X，其实需要 Y"的典型场景？
你的背景：[简要说明你在这个领域的经验]
```

### 第二步：按子任务执行

#### T1.1 — 描述你领域中的"找"的痛点
**做什么**：写出你领域中"找人/找服务/找方案"的 3 个典型场景。每个场景说清楚：谁在找什么？现在怎么找的？痛点在哪？
**交付**：3 个场景描述（每个 200-300 字）

#### T1.2 — 绘制用户旅程
**做什么**：选 1 个最典型的场景，画出完整的用户旅程（从"产生需要"到"问题解决"）。标注情绪、痛点、决策点。
**交付**：用户旅程图（形式不限）

#### T1.3 — 建立数据模型
**做什么**：在你的场景中，做好匹配/连接需要什么信息？分"硬性"和"软性"两类。哪些信息目前难以获取？
**交付**：数据模型文档

#### T1.4 — 创建 3 个典型画像
**做什么**：设计 3 个代表性的用户画像。每个画像包含认知转变路径——用户"以为需要什么"vs"实际需要什么"。
**交付**：3 个画像卡片

#### T1.5 — 响应范式 vs 搜索范式对比
**做什么**：在你的具体场景中，"发出信号等响应"和"按条件搜索过滤"各自的优劣是什么？有没有搜索更好的情况？
**交付**：对比分析（500-1000 字）

## 我们提供什么

### 概念着陆（通爻术语 → 你领域中的等价表达）

做这个任务不需要理解通爻的技术。你需要理解的核心概念，用生活语言翻译如下：

| 通爻概念 | 在你领域中可能的等价表达 | 你在建模中要做的事 |
|---------|---------------------|-------------------|
| 投影 | 一个人在不同场景中展现的不同面。比如同一个厨师，在家宴和商业宴会中展现的技能和气质完全不同 | 识别你领域中的"投影差异"——同一个人在什么情况下展现不同面？ |
| 共振 | 遇到"对的人/对的服务"时的感觉。不只是"条件匹配"，还有一种"对味"的感觉 | 定义你领域中的"对味"到底包含什么维度？技能只是其中一部分 |
| 响应范式 | 发出需求让合适的人主动来，而非自己去找。比如在朋友圈发一条"有人认识靠谱的装修队吗？"vs 自己在平台上一个个看评价 | 分析你领域中"发朋友圈问"比"自己搜索"好在哪？反过来呢？ |
| 认知转变 | "以为需要 X，发现其实需要 Y"。比如以为需要"一个好厨师"，其实需要的是"一个理解我口味偏好的菜单方案" | 找到你领域中的典型认知转变故事——这是最有价值的发现 |
| 需求 ≠ 要求 | 需求是内心的张力（"想吃得健康"），要求是自己假设的解法（"找一个素食餐厅"）。解法可能是错的 | 区分用户说的（要求）和用户真正需要的（需求） |

### 设计原则（与场景建模的关系）

| # | 原则 | 在场景建模中的意义 |
|---|------|-------------------|
| 0.6 | 需求 ≠ 要求 | 你的建模需要识别用户"说出来的要求"和"真正的需求"之间的差距 |
| 0.8 | 投影是基本操作 | 同一个人在不同场景中的"投影"不同——你的数据模型需要捕捉这种差异 |
| 0.7 | 简单规则涌现复杂行为 | 好的场景建模不是列出100条规则，而是找到几个核心维度 |

## 做完了是什么样

### 产出清单

1. **3 个场景描述**（每个 200-300 字）— 你领域中"找人/找服务"的典型痛点
2. **1 个用户旅程图** — 从"产生需要"到"问题解决"的完整路径
3. **数据模型文档** — 做好匹配需要什么信息
4. **3 个典型画像** — 含认知转变路径
5. **响应范式 vs 搜索范式对比** — 在你的场景中的分析

### 三级质量标准

**做完了（基本合格）**：
- 场景基于真实经验或调研，不是凭空想象
- 用户旅程标注了情绪和决策点
- 数据模型区分了"硬性信息"和"软性信息"

**做得好（超出预期）**：
- 发现了至少 1 个"以为需要 X，其实需要 Y"的真实案例
- 响应范式 vs 搜索范式的对比是诚实的——指出了搜索更好的情况
- 画像有足够细节，让从未接触过这个领域的人也能理解

**做得出色（产生额外价值）**：
- 场景建模的框架可以被其他领域的共建者直接参考——成为 T1 的"标杆实例"
- 发现了你领域中响应范式的独特优势——这是其他领域不具备的
- 提出了具体的产品设计建议——不只是"理解用户"，还指出了"应该怎么做"
- 用你领域的真实故事验证了"认知转变"概念的普适性
