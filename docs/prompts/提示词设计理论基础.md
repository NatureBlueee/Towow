# ToWow 提示词设计理论基础

> 这份文档为提示词设计提供心理学和博弈论的理论支撑。

---

## 一、预测"谁会参与"的心理学模型

### 1.1 任务本质的定义

这个任务**不是**"精准预测谁会参与"，**而是**：

> 设计一个筛选机制，在几百个Agent中识别出"值得询问"的候选人。

"值得询问"意味着：
- 问了之后有合理概率得到积极回应
- 即使被拒绝，拒绝的理由也是有信息量的（而不是"完全不相关"）

这是一个**信号识别问题**——从有限的信息中识别出"可能性信号"。

### 1.2 六层心理决策模型

当一个人收到"要不要参与某件事"的邀请时，他的心理过程：

#### 第一层：注意力门槛（Attention Gate）

- 这件事进入我的意识了吗？
- 我有没有心理带宽去处理这个信息？
- 如果我正在忙别的事情，这个邀请可能直接被过滤掉

**影响因素**：当前忙碌程度、心理带宽、注意力焦点

#### 第二层：相关性判断（Relevance Assessment）

这件事和我有什么关系？

- **能力相关**：我能帮上忙吗？
- **身份相关**：这是"我这种人"会参与的事吗？
- **关系相关**：发起者是谁？我认识吗？想认识吗？
- **兴趣相关**：这是我最近关心的领域吗？

**影响因素**：能力画像、身份认同、社交网络、兴趣领域

#### 第三层：成本-收益的直觉评估（Intuitive Cost-Benefit）

**注意：这不是理性计算，而是"感觉"。**

- 参与这件事"感觉"费劲吗？（时间、精力、社交成本）
- 参与这件事"感觉"有价值吗？（有趣、有用、有意义、有社交收益）

这个"感觉"受到当前状态的强烈影响——心情好的时候门槛低。

**影响因素**：当前情绪、精力状态、近期体验

#### 第四层：风险与承诺感知（Risk & Commitment Perception）

- 参与这件事会不会让我陷入麻烦？
- 我需要承诺多少？能不能中途退出？
- 如果搞砸了，对我有什么影响？

**影响因素**：风险偏好、承诺风格、历史经验

#### 第五层：社会性考量（Social Considerations）

- 其他谁在参与？
- 参与这件事会让我在社交网络中获得什么？
- 不参与会不会有社交成本？（比如拒绝朋友的邀请）

**影响因素**：社交网络、社交需求、社会身份

#### 第六层：决策惯性与默认选项（Decision Inertia）

- 我的默认倾向是什么？（有的人默认说"是"，有的人默认说"否"）
- 需要多大的信号强度才能让我偏离默认？

**影响因素**：人格特质、决策风格

### 1.3 可观测信号与心理状态的映射

| 心理层面 | 可观测程度 | 可用的外部信号 |
|---------|-----------|--------------|
| 注意力门槛 | 中等 | 当前忙碌程度、最近活跃度、时间偏好 |
| 相关性判断 | **较高** | 能力描述、身份标签、兴趣领域、社交关系 |
| 成本-收益直觉 | 低 | 只能通过历史行为模式推断 |
| 风险承诺感知 | 低 | 协作风格偏好（如果有记录的话） |
| 社会性考量 | 中等 | 社交网络关系、过往协作记录 |
| 决策惯性 | 中等 | 历史接受/拒绝比例、人格特征 |

**关键洞察**：相关性判断是筛选的核心，因为可观测性最高。但不能忽视当前状态对决策的影响。

### 1.4 给团队的任务描述

---

**任务：设计Agent筛选的心理学模型**

**背景**：
ToWow网络中有几百个Agent（代表真实用户）。当一个需求广播时，中心Agent需要从中筛选出10-20个"值得邀请"的候选人。

**核心问题**：
1. 人决定是否响应一个合作邀请，经历哪些心理阶段？
2. 每个阶段的关键影响因素是什么？
3. 哪些因素可以通过Agent简介信息来推断？
4. 筛选算法应该优先考虑哪些因素？权重如何设定？

**期望产出**：
1. 一个分层的心理决策模型（初稿已有，见上）
2. 每层需要的信息字段定义
3. 筛选提示词的理论基础说明

**约束条件**：
- 这是MVP，不追求精准预测，目标是"不遗漏明显相关的人"
- 宁可多邀请几个，不要漏掉关键人选
- 信息获取成本要考虑（不能要求用户填写太多字段）

## 二、SecondMe的认知锚点设计

### 2.1 问题本质

SecondMe的任务是：代表用户决定是否参与一个合作机会。

它的默认倾向是"乐观"——倾向于找到连接、接受合作。

**这个倾向是对的**（产品设计目标是帮用户建立连接），但需要"校准"，让乐观建立在准确理解的基础上。

核心问题：
> 怎么让SecondMe在保持乐观倾向的同时，更准确地模拟这个具体用户的决策模式？

### 2.2 认知锚点的作用

认知锚点给SecondMe提供一个"参照系"，让它在评估时有据可依，而不是凭直觉（模型的直觉往往是训练数据的平均值）。

好的认知锚点应该：
1. **具体**：不是抽象原则，而是可以直接对照的检查项
2. **有区分度**：能帮助区分"这个用户会/不会"，而不是"一般人会/不会"
3. **可验证**：SecondMe可以基于用户记忆来回答这些检查项

### 2.3 三层认知锚点框架

#### 锚点层1：用户的决策风格（稳定特征）

描述用户是什么样的决策者，相对稳定，不因具体事件变化。

| 维度 | 低 | 高 |
|------|-----|-----|
| **启动门槛** | 好奇心驱动，容易尝试新事物 | 谨慎型，需要充分理由 |
| **承诺风格** | 轻承诺，愿意先答应再说 | 重承诺，答应了就一定做到 |
| **社交能量** | 外向，社交活动给他充电 | 内向，社交活动消耗他 |
| **风险偏好** | 冒险型，愿意尝试不确定的事 | 稳健型，偏好确定性 |
| **时间敏感度** | 时间充裕感强 | 时间紧迫感强 |

#### 锚点层2：用户的当前状态（动态特征）

描述用户现在的状态，会影响他对同一件事的不同反应。

| 维度 | 需要了解的信息 |
|------|--------------|
| **当前忙碌程度** | 这周/这个月在忙什么？有多少空闲心理带宽？ |
| **当前关注焦点** | 最近在关心什么话题/领域？ |
| **当前情绪基调** | 最近心情如何？精力状态如何？ |
| **当前社交需求** | 最近是想多认识人，还是想安静独处？ |
| **近期承诺负担** | 最近已经答应了多少事情？ |

#### 锚点层3：与具体需求的匹配度（情境特征）

针对当前这个具体合作邀请的评估。

| 维度 | 评估问题 |
|------|---------|
| **能力匹配** | 用户能提供这个需求需要的东西吗？ |
| **兴趣匹配** | 这个需求涉及的领域/话题是用户感兴趣的吗？ |
| **关系匹配** | 用户和需求发起者有什么关系？认识/不认识/想认识？ |
| **成本感知** | 参与这件事对用户来说"费劲"吗？ |
| **收益感知** | 用户能从参与中得到什么？ |
| **风险感知** | 参与这件事有什么潜在负面后果？ |

### 2.4 平衡乐观与准确的机制

**乐观和准确不是对立的。**

- **乐观**体现在：愿意看到潜在连接，愿意给机会一个"试试看"的空间
- **准确**体现在：对用户的状态和偏好有正确理解，不强行匹配

具体机制：

1. **先充分分析，再做决策**——不要跳过分析直接得结论
2. **允许"弱参与"选项**——不是只有"参与/不参与"，可以是"感兴趣但有顾虑"、"需要更多信息"
3. **显性表达不确定性**——如果SecondMe不确定用户会怎么想，应该说出来，而不是猜一个
4. **区分"能力匹配"和"意愿匹配"**——能帮上忙不等于想帮

## 三、冲突处理的博弈分析

### 3.1 博弈结构

当用户需求和参与者能力/意愿出现冲突时，这是一个**多方博弈**场景：

- **需求方**：想要某个结果，有自己的约束条件（预算、时间、要求）
- **供给方（多个）**：各自有不同的能力、意愿、条件
- **系统（ToWow）**：不是博弈参与者，而是**信息中介**和**方案建议者**

### 3.2 系统的角色定位

**系统不是决策者，而是信息呈现者 + 可能性分析者。**

#### 系统应该做的事

1. **呈现事实**：现在的情况是什么，找到了什么，没找到什么
2. **分析可能性**：如果调整某个条件，可能会出现什么结果
3. **识别博弈空间**：哪些条件是硬约束（不可协商），哪些是软约束（有谈判空间）
4. **提示用户选择**：把决策权交给用户

#### 系统不应该做的事

- ❌ 替用户决定接受某个妥协
- ❌ 压着参与者接受用户的要求
- ❌ 假装冲突不存在
- ❌ 单方面偏向某一方

### 3.3 冲突类型与处理模式

#### 类型A：资源不足型冲突

**场景**：用户要100人场地，只找到20人场地。

**处理模式**：
```
【当前情况】
- 您需要：100人规模的活动场地
- 我们找到：1个20人场地（来自agent_bob）

【为什么会这样】
- 在北京地区的Agent中，只有agent_bob表示可以提供场地
- 100人规模的场地确实稀缺

【可能的方向】
- 如果您考虑分批举办，agent_bob的场地可以满足
- 如果坚持100人规模，可能需要动用其他资源
- 可以继续扩大搜索范围

【您的选择】
- 是否要和agent_bob进一步沟通？
- 是否要调整需求后重新搜索？
```

#### 类型B：条件冲突型

**场景**：用户要免费场地，唯一的场地方要收费。

**处理模式**：
```
【当前情况】
- 您的期望：免费场地
- agent_bob的条件：需要收费（具体费用可协商）

【博弈空间分析】
- "免费"是您的硬性要求，还是有弹性？
- agent_bob表示费用"可协商"——有谈判空间
- 可能的交换筹码：活动宣传露出、长期合作、资源置换

【您的选择】
- 是否要和agent_bob直接沟通，看看能否达成置换方案？
- 您有什么可以提供的价值？
```

#### 类型C：时间紧迫型冲突

**场景**：用户要明天的活动，参与者说时间太赶。

**处理模式**：
```
【当前情况】
- 您的需求：明天举办活动
- 参与者反馈：3位表示时间太赶，1位表示无法参与

【时间约束分析】
- "明天"是硬性要求吗？
- 如果推迟2-3天，3位参与者都可以参加
- 如果必须是明天，目前只有1位确认

【您的选择】
- 时间是否有调整空间？
- 如果必须是明天，是否接受缩小规模？
```

### 3.4 博弈分析的设计原则

1. **先呈现，后分析，最后给选项**——不要跳步
2. **明确区分硬约束和软约束**——硬约束不可协商，软约束有空间
3. **识别各方的真实利益**——不只是表面要求
4. **寻找共赢空间**——有没有双方都能接受的中间方案
5. **把决策权交给用户**——系统是顾问，不是决策者

## 四、"收获感"的设计

### 4.1 收获感的本质

用户使用ToWow后，即使没有得到完美匹配，也应该感受到：

1. **被认真对待**：系统真的在帮我找，不是敷衍
2. **信息增量**：我知道了一些之前不知道的事情
3. **可能性被展开**：即使这次不成，我知道了还有什么可能
4. **下一步清晰**：我知道接下来可以做什么

### 4.2 收获感的设计要素

#### 要素1：过程可见

让用户看到系统做了什么：
```
【搜索过程】
- 分析您的需求：识别出3个关键能力需求
- 在北京地区搜索了87位Agent
- 初步筛选出23位可能相关的Agent
- 发出邀请，收到12个回应
```

#### 要素2：发现要展示

即使不完美匹配，也要展示找到了什么：
```
【我们找到的】
虽然没有找到完全满足您需求的方案，但我们发现了：
- 3位Agent表示有场地资源（但规模不完全匹配）
- 2位Agent表示可以做嘉宾分享
- 1位Agent表示愿意帮忙推广

这些发现可能对您有参考价值。
```

#### 要素3：原因要解释

不是简单说"没找到"，而是说清楚为什么：
```
【为什么难以匹配】
- 100人规模的场地确实稀缺
- "免费"的期望进一步缩小了范围
- 2月15日是周末，很多场地已被预订
```

#### 要素4：替代路径要指出

给用户下一步的方向：
```
【您可以考虑的方向】
1. 调整规模预期：50人以内选择会多很多
2. 动用自己资源：您是否有可以置换的资源？
3. 分批举办：同样内容办2-3场
4. 扩大搜索：更广的地理范围

需要我按某个方向重新搜索吗？
```

### 4.3 最终输出的标准结构

```
## 📋 需求理解
[一句话复述用户的核心需求]

## 🔍 搜索过程
[系统做了什么，搜索了多少范围]

## ✅ 匹配结果
[详细展示方案，或说明匹配程度和差距]

## 📊 发现与洞察
[即使没有完美匹配，也展示找到了什么有价值的信息]

## ❓ 未能满足的部分
[诚实说明哪些需求没有被满足，以及原因]

## 🚀 下一步建议
[2-3个可行的选项，让用户选择]
```

### 4.4 关键原则

1. **永远不要返回空结果**——即使没有完美匹配，也要返回"我们尝试了什么、发现了什么"
2. **过程透明比结果重要**——用户要感受到系统在认真工作
3. **失败是有信息量的**——告诉用户为什么难以匹配，本身就是有价值的信息
4. **决策权在用户**——系统提供选项和分析，不替用户决定
