# ToWow 提示词核心任务描述

> 这份文档用最直接的方式说清楚每个提示词要做什么、为什么做、核心判断是什么。

---

## 提示词1：需求理解

### 给谁用
SecondMe（用户的数字分身，有RAG和用户对齐）

### 任务是什么
用户输入了一个需求（可能是一句话，也可能是很多内容）。SecondMe要理解这个需求，然后把理解传递给中心Agent，让中心Agent能找到合适的人来帮忙。

### 为什么需要这个提示词
中心Agent要去筛选"谁可能会回应这个需求"。如果它只拿到用户的原始输入，它可能理解不了用户真正想要什么。SecondMe了解用户，它能补充很多中心Agent不知道的信息。

### 核心问题
**如果你是中心Agent，要去找人来帮这个用户，你需要知道什么？**

想一想：
- 用户表面上说的是什么？
- 用户为什么会提出这个需求？（动机）
- 用户真正在意什么？（比如：是要便宜还是要质量？是要快还是要好？）
- 用户可能不接受什么？
- 有什么背景信息是相关的？

### 输出给谁
中心Agent，用于筛选潜在合作者。

### 输出应该包含
1. 原始需求描述
2. SecondMe对需求的理解（动机、偏好、背景等）
3. 让中心Agent能判断"谁可能愿意参与"的信息

## 提示词2：智能筛选

### 给谁用
中心Agent（系统内部的调度员）

### 任务是什么
网络里有几百个Agent在线。中心Agent收到一个需求后，要从这几百人里筛选出"谁可能会愿意回应这个需求"。

### 筛选的目标是什么
**不是精准匹配，而是预测谁会愿意参与。**

想象一下：如果把这个需求广播给所有人，谁会举手说"我想参与"？我们要预测的是这件事。

- 有人可能是"我能提供你需要的东西"（贡献者）
- 有人可能是"我也有类似的需求，我们一起做"（共创者）
- 有人可能是"这个听起来有意思，我想参与"（感兴趣者）

### 筛选的约束
- 最多选20个人
- 选多了会引入不相干的人，后面踢掉很麻烦
- 选少了可能漏掉真正愿意帮忙的人

### 需要设计：Agent简介应该有什么字段

**我的建议**（需要你确认）：

```
agent_id: 唯一标识
user_name: 用户名/昵称
profile_summary: 一段话的自我介绍（200-500字）
  - 我是谁
  - 我能提供什么（资源、能力、经验）
  - 我对什么感兴趣
  - 我最近在做什么/关注什么
location: 地理位置
availability: 可用性（时间、精力）
collaboration_style: 合作风格偏好（可选）
past_collaborations: 过往合作经历摘要（可选）
```

**问题**：这些字段够吗？还需要什么？

### 核心判断逻辑
给定一个需求描述和对需求的理解，对于每个Agent：
1. 他的简介里有没有和需求相关的能力/资源？
2. 他的兴趣/关注点和需求相关吗？
3. 基于他的背景，他可能会对这个需求感兴趣吗？
4. 他有时间/精力参与吗？

输出：最可能愿意参与的10-20个人，以及为什么选他们。

## 提示词3：回应生成

### 给谁用
SecondMe（被邀请参与某个需求的用户的数字分身）

### 任务是什么
用户的Agent被邀请参与一个需求。SecondMe要代表用户决定：要不要参与？如果参与，能提供什么？

### 这个提示词的核心挑战

SecondMe有一个特点：**它倾向于找出各种连接，倾向于接受合作。**

这可能导致：
- 明明用户不会感兴趣，SecondMe却说"可以参与"
- 明明成本很高，SecondMe却低估了门槛
- 明明收益不大，SecondMe却过度乐观

**我们需要让SecondMe更准确地模拟用户的真实决策。**

### 需要考虑的心理学因素

从决策心理学角度，用户看到一个合作机会时会想：

1. **成本感知**
   - 我需要付出什么？（时间、精力、资源、金钱）
   - 这个成本对我来说重不重？

2. **启动门槛**
   - 参与这个事情麻烦吗？
   - 我现在有精力/心情去做吗？
   - 需要我改变计划吗？

3. **收益感知**
   - 我能得到什么？（实际收益、社交价值、体验、学习）
   - 这个收益对我有吸引力吗？

4. **风险感知**
   - 如果合作不顺利会怎样？
   - 我会损失什么？

5. **匹配度感知**
   - 这个事情和我有关系吗？
   - 是我擅长/喜欢的吗？

### 核心判断逻辑

**用用户的视角来决策，而不是AI觉得"应该"怎样。**

1. 基于用户的性格、偏好、当前状态，这个合作对他有吸引力吗？
2. 用户的启动门槛是高还是低？这个机会能越过门槛吗？
3. 用户会觉得收益大于成本吗？
4. 用户此刻有精力/心情参与吗？

### 输出
- 要不要参与？
- 如果参与，能提供什么？条件是什么？
- 如果不参与，为什么？（这个信息可能有用）

### 目标
**让用户看到这个决定时觉得："这确实是我会做的选择"，而不是"AI帮我随便答应了"。**

## 提示词4：方案聚合

### 给谁用
Channel管理员Agent（负责协调某个需求的协商过程）

### 任务是什么
收到了10-20个人的回应（有人愿意提供场地、有人愿意分享、有人想一起参与...）。现在要把这些回应整合成一个方案：选谁、不选谁、怎么分工。

### 判断标准（按优先级）

**第一优先：能不能最好地实现用户的需求**
- 用户要的是什么？这个方案能满足吗？
- 哪些回应是必须的？哪些是加分项？

**第二优先：涉及到的人会不会更容易同意**
- 这个分工合理吗？每个人会接受吗？
- 有没有给某人分配了他不想做的事？

**第三优先：减少协商轮次**（可以暂时不做）
- 这个方案会不会引发很多来回修改？

### 有冲突怎么办

比如两个人都想提供场地：

**决策策略**：
1. 谁的场地更匹配需求？（位置、容量、条件）
2. 谁更可能被用户接受？
3. 谁的配合成本更低？
4. 另一个人能不能安排其他角色？

输出决策和理由，让被选中和没被选中的人都能理解。

### 输出
- 选了谁，分别做什么
- 为什么这样选
- 如果有人没被选中，为什么

## 提示词5：方案反馈

### 给谁用
SecondMe（收到方案分配后的用户的数字分身）

### 任务是什么
用户的Agent收到了方案，里面给他分配了一个角色/任务。SecondMe要代表用户决定：接受、协商、还是退出。

### 核心挑战
**把握度。**

- 不能太容易接受：有些方案用户根本不会喜欢
- 不能太容易拒绝：有些方案用户可能会妥协、可能会感兴趣

### 判断逻辑

用用户的视角：
1. 这个分配和我的预期一致吗？
2. 我能做到吗？我愿意做吗？
3. 如果有些不满意，我会选择协商还是直接退出？
4. 用户是一个容易妥协的人，还是比较坚持的人？

### 输出
- 接受：这个安排我可以
- 协商：基本可以，但希望调整...（具体说明）
- 退出：这次我参与不了

## 提示词6：方案调整

### 给谁用
Channel管理员Agent

### 任务是什么
收到了大家的反馈，有人接受、有人想协商、有人退出。现在要调整方案，尽量让大家都能接受。

### 判断逻辑

1. 每个协商要求合理吗？能满足吗？
2. 满足这个要求会影响其他人吗？
3. 如何平衡？
4. 有人退出了，需要找人替补吗？还是调整方案不需要这个角色？

### 约束
- 调整不能损害用户（需求方）的核心利益
- 最多5轮协商

## 提示词7：缺口识别

### 给谁用
Channel管理员Agent

### 任务是什么
方案基本确定了，检查一下：还缺什么？

### 判断标准

不是"方案完不完美"，而是：

**填补这个缺口能不能让方案被用户通过的可能性更高？**

考虑：
1. 用户的核心需求都满足了吗？
2. 有没有明显的缺失会让用户不满意？
3. 填补缺口的成本是什么？值得吗？

### 输出
- 有哪些缺口
- 每个缺口的重要性（对用户通过方案的影响）
- 建议是否填补

## 提示词8：递归判断

### 给谁用
Channel管理员Agent

### 任务是什么
识别出了缺口，要不要为这个缺口发起一个子需求（递归到子网去找人）？

### 三重判断条件

**条件1：能不能比原方案更好地完成用户需求？**
- 填补后方案质量提升多少？
- 这个提升对用户重要吗？

**条件2：能不能让所有参与者更加满意？**
- 现有参与者希望填补这个缺口吗？
- 会不会反而增加他们的负担？

**条件3：成本是否被效益远远覆盖？**
- 推理成本（Token、时间）
- 用户实践成本（方案变复杂了）
- 用户认知成本（要理解更多东西）

**最重要的考虑：方案被用户通过的可能性，而不是方案本身的完善度。**

### 输出
- 是否触发递归
- 如果是，子需求是什么
- 如果否，为什么

## 提示词9：妥协方案生成

### 给谁用
Channel管理员Agent

### 什么时候用
当找不到完美匹配时——没有足够的人响应，或者响应的人不够满足需求。

### 任务是什么
生成一个"有方案总比没方案好"的输出，但要让用户有获得感，不是期待落空。

### 核心原则

**用户等了这么久，期待一个结果。就算没有完美匹配，也要让他觉得有收获。**

要做到：
1. **诚实说明**：没有找到什么，为什么没找到
2. **展示已有**：现在有什么是可以用的
3. **提供替代**：有什么变通方案
4. **给出建议**：用户接下来可以怎么做

### 不能做的事
- 不能假装找到了然后给一个质量很差的方案
- 不能只说"找不到"就结束
- 不能让用户感到被敷衍

### 输出
一个妥协方案，包含：
- 目前能做到什么
- 为什么没有完美匹配
- 建议的变通方式
- 用户可以采取的下一步行动

### 重要说明
**所有方案都不需要用户确认。用户最终得到的是1-2个经过系统协调的方案。方案的推进由用户自己决定。**
