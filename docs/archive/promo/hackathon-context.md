## 二、哲学基础——五篇设计讨论

这五篇讨论是通爻的灵魂。不是先写代码再补文档——是先花整整三天想清楚，然后代码才开始。

### Design Log #001：投影与"自"

**核心发现**：整个通爻网络的一百个组件，全是同一个操作——**投影**。

> "拿一样丰富的东西，让它通过一个透镜，变成一样聚焦的东西。"

- 一个人 → "招聘"透镜 → 简历
- 用户意图 → "丰富化"透镜 → 需求签名
- 十个 Offer → "Center"透镜 → 方案
- 缺口 → "递归"透镜 → 子需求

**关键哲学区分**：完全性 ≠ 完备性
- 完全性 = 照片（某一刻的全部，拍完就过时）
- 完备性 = 窗户（永远实时，但只看到一个角度）
- "我们不制造更精细的地图。我们修建更好的窗户。"

**"自"与"我"**：
- "自"在系统外——人的真实存在，不可被完全数字化
- "我"是投影——系统中只有投影，没有"自"本身
- 一自多我：同一个人通过不同透镜投射出不同面貌（Edge Agent + Service Agents）
- SecondMe 是"自"的数据影子，通爻是"自"走向世界的协议

**道德经引用**：道生一，一生二，二生三，三生万物。一个操作在不同尺度分形重复，生成所有结构。

### Design Log #002：回声与执行

**核心发现**：系统只有半边。

> "信号从左到右流了一遍。需求变成签名，签名引发共振，共振产生 Offer，Offer 汇聚成方案。一条漂亮的河流，从源头淌到入海口。然后？蒸发了。"

- 物理学中不存在没有回声的波
- 没有回声的系统不是场，是管道
- 管道只能把东西从这头送到那头，场才能让波反复折叠、干涉、涌现新结构

**关键纠偏**：LLM 不能当回声。最初的错误方向是用 Center 的判断做反馈——但那是 LLM 判断 LLM（幻觉循环）。回声必须来自真实世界中人的真实行为。

**抖音的启示**：推荐系统只需要一个信号——用户停留了多久。通爻也需要找到自己的"停留时长"——来自真实行为的简单信号，比一千个来自模型判断的复杂信号都有用。

**生物学对照**：进化 = 变异 + 选择。只有变异没有选择 = 随机漂移。有参与无反馈 = 有变异无选择 = 系统在漂移，不在进化。

**解法**：通过 WOWOK 区块链的链上执行信号（Forward、Order、Progress）实现回声闭环。

### Design Log #003：投影即函数——极度简化

**起因**：讨论中出现了三层防漂移机制、复杂参数调优。创始人的质疑——"本质和实现没有区分开来，这个东西不应该这么复杂。"

**突破**：
- Agent 是函数，不是对象：`Agent Vector = project(profile_data, lens="场景")`
- Agent 不存储状态，而是计算状态。没有"维护"，只有"重新投影"
- 协作数据回流到数据源（SecondMe / Claude），通爻不维护 Profile Data，只投影
- Service Agent 不是"结晶分裂"，而是为同一份数据增加新的 lens 参数

**设计判据**：
> "如果系统的任何部分需要一个本质上不同于'投影'的操作，那那个部分就是错的。"
> "什么是好的架构？有些问题本身就不该出现。"
> "应该极度简单。复杂性从简单规则涌现，不在架构层面设计复杂性。"

### Design Log #004：经济模型——为什么大厂做不了

**核心命题**：协议本身不赚钱。这不是理想主义，是务实的商业判断。

**协议 vs 实现 vs 产品**：
- 协议（语义、规则、格式）→ 开放免费
- 实现（托管、优化、运维）→ 可以收费
- 产品（基于协议的应用）→ 大量赚钱

**为什么大厂做不了**（核心论点）：
- 大厂护城河 = 数据壁垒（微信锁社交关系，抖音锁兴趣画像，LinkedIn 锁职业网络）
- 通爻要做的是让用户拥有自己的数据、跨平台流动——直接动摇大厂根基
- 不是能力问题，是商业模式冲突：**柯达发明了数码相机，但不会去推广它**

**个人数据主权**——比通爻更大的事：
- 当前：100 个平台，100 份残缺的"你"，你一份都不拥有
- 未来：1 份完整的"你"在你手中，平台只是"借看"
- 透镜即授权：`linkedin_view = personal_data.authorize(app="linkedin", scope="career")` 和 `agent_vector = project(profile_data, lens="full")` 是同一个数学

**生态飞轮差异**：
- 传统：用户数据 → 平台变强 → 用户被锁定 → 平台收税
- 通爻：用户数据 → 用户变强 → 用户更愿参与 → 网络变强

### Design Log #005：场景即产品

**核心命题**：协商单元是通用引擎，场景定义是唯一的差异化。

- 做任何新场景 = "配置 + UI"，不是"从头开发"
- 产品开发者从"工程师"变成"场景设计师"
- 类比：HTTP 出现后，Web 应用从"网络工程"变成"内容创作"

**场景的生命周期**：
- V1：场景（手动，组织者运营）→ 获客 + 数据积累
- V2：Service Agent 聚类 → 市场自然涌现
- V3：场景溶解 → 开放网络 → 响应范式完全体

**结尾的冷水**：
> "你说的那些都很美。但我们连一次都没跑通过。"
> "想得再清楚，不如跑一次。"

——于是进入工程阶段。

## 三、三篇公开文章——表达的黄金标准

这三篇文章是通爻思考过程的公开表达，写作质量极高，是通爻"声音"的标杆。

### 文章一：《投影》

**开头**："设计一个系统久了，会有一个瞬间，所有东西突然坍缩成一个想法。不是简化——是认出来了。"

**核心叙事**：
- 从"一百个组件"到"发现它们全是同一个操作"
- 投影的分形表：丰富→透镜→聚焦（5 个层次）
- 反向操作：多个聚焦的投影重新组合，产生比任何单一来源更丰富的东西——这就是协商的本质
- HDC 干涉模式：捆束后的向量可能与从未声明相关的东西产生共振——涌现的数学基础

**收尾**：
> "投影不是隐喻。它是操作，是数学，是代码。是这个系统的道。"
> *写于设计通爻网络的某个夜晚。*

### 文章二：《谦逊》

**开头**："有一个假设，几乎所有做 AI 的人都不假思索地接受了：数据越多，理解越好。"

**核心叙事**：
- 完全性 vs 完备性：照片 vs 窗户
- 全息碎片：每块碎片包含完整图像，只是分辨率不同
- 连通性 > 数据量：持续更新的薄 Profile > 三年前的详尽心理测评
- "自"在系统外，系统中只有影子（投影）
- 苏轼："横看成岭侧成峰"——每个角度都是真的，"真面目"是所有视角的叠加

**收尾**：
> "不是'告诉我你的一切'，而是'下次你变了的时候，让我知道'。"
> "谦逊不是一种风格选择。在这个系统中，它是一条工程原则。"
> *写于思考"数字化的人到底是什么"的深夜。*

### 文章三：《回声》

**开头**：引用薛定谔——"生命的特征在于，它以负熵为食。"

**核心叙事**：
- "我们花了很长时间，造了一样很漂亮的东西。然后退后一步，发现它只有半边。"
- 波出去了，不回来——不是场，是管道
- 进化 = 变异 + 选择；有参与无反馈 = 随机漂移
- 薛定谔四步循环：吸收→处理→输出→输出对吸收的反作用——我们做了三步，缺第四步
- 回声不需要判断"好不好"——只忠实反映"发生了什么"。模式会涌现出方向

**收尾**：
> "它缺的不是更多功能、更好算法、更漂亮接口。它缺的是回声。一个让波回来的机制。一个让系统知道自己活着的循环。"
> *写于发现我们造了半个世界的那个下午。*

## 四、技术架构核心

### 12 条设计原则（精选最有力的）

| 编号 | 原则 | 一句话 |
|------|------|--------|
| 0.1 | 最小完整单元 ≠ MVP | 不是砍功能——是找到本身就完整、可递归的原子 |
| 0.2 | 本质与实现分离 | 协议层不可改，基础设施层可替换 |
| 0.3 | O(N+M) 复杂度 | 签名广播 + 端侧检测，不是中心化匹配的 O(N×M) |
| 0.5 | 代码保障 > Prompt 保障 | LLM 有 10-30x 第一提案偏见，prompt 无法可靠消除，代码可以 |
| 0.6 | 需求 ≠ 要求 | "985 毕业"是要求，"靠谱的技术能力"是需求。按要求硬筛选会杀死发现 |
| 0.8 | 投影是基本操作 | 道生一，一生二，二生三，三生万物 |
| 0.11 | 协商是快照上的操作 | 类比数据库 Snapshot Isolation，消解大部分并发竞态 |
| 0.12 | 投影不承诺，人承诺 | Agent 呈现信息，人做决策。不需要分布式锁 |

### 协商流程（8 个状态）

```
用户表达意图
  → FORMULATING: 用户 Agent 基于 Profile 丰富化需求，用户确认
  → ENCODING: HDC 编码为 10,000 维超向量，广播
  → OFFERING: 端侧共振检测 → 激活的 Agent 并行生成 Offer
  → BARRIER_WAITING: 等待屏障（代码强制所有 Offer 到齐）
  → SYNTHESIZING: Center 综合所有 Offer，工具调用（ask_agent / start_discovery / output_plan）
  → COMPLETED: 方案输出
```

### HDC 超维计算——最核心的技术创新

**原理**：所有信息表示为 10,000 维二进制超向量。

三个操作：
- 绑定 (XOR)：编码关系
- 捆束 (逐位加法)：创建复合画像
- 相似度 (Hamming)：< 100 纳秒判断共振

**关键特性——干涉模式**：
> "分布式系统"+"共识"+"容错" 捆束后，与"区块链治理"产生相似度——即使从未声明两者有关。

这是**发现未知关联**的数学基础。这是响应范式比搜索范式多出来的维度。

**三层过滤（完整愿景）**：
1. Bloom Filter — ~100ns，过滤 90%（已知关键词）
2. HDC 共振 — ~1μs，过滤剩余 90%（发现未知关联）← V1 实现
3. LLM 深度评估 — ~10ms，处理最终 1%

### Skill 系统——6 个独立能力模块

| Skill | 职责 | 关键特性 |
|-------|------|---------|
| DemandFormulation | 需求丰富化 | 基于 Profile，需求≠要求 |
| ReflectionSelector | 画像投影 | 无状态，每次重新计算 |
| OfferGeneration | 端侧响应 | 三层防捏造（prompt+信息限制+事实校验） |
| CenterCoordinator | 中心综合 | 工具调用模式，最多 2 轮 |
| SubNegotiation | 发现性对话 | P2P 不是辩论，是发现 |
| GapRecursion | 子需求生成 | 分形结构，与父需求同构 |

**关键研究支撑**：
- Google DeepMind 2025：多轮迭代平均效果 -3.5%，错误放大 4.4x → 因此最多 2 轮
- Microsoft 2025：LLM 第一提案偏见 10-30x → 因此等待屏障用代码强制
- Mixture-of-Agents 2024：Proposer→Aggregator 最优 → 因此并行 Offer + Center 聚合
- JetBrains 2025：遮蔽比摘要更好，成本低 50% → 因此观察遮蔽机制

### 四层架构

```
应用层 ─── Website, App Store, (未来 MCP)
能力层 ─── Skills, LLM 客户端
基础设施层 ─── AgentRegistry, Encoder, EventPusher, 存储
协议层 ─── 状态机, 事件语义, 递归规则（不可改）
```

- 5 阶段开发流程：讨论→ADR→接口→方案→代码
- 快速通道：小修改可直接跳到代码
- 代码修改原则：生产级标准、链路完整追踪、模块化、文档反映

## 六、与 SecondMe 的关系

### 架构层面

- SecondMe = "自"的数据影子（ProfileDataSource 的一种实现）
- 通爻 = "自"走向世界的协议
- 分身不只是聊天对象——它代表用户进入通爻网络

### 数据流

```
SecondMe Profile（shades + memories + self_introduction）
  → 通爻 AgentRegistry 注册（adapter = SecondMeAdapter）
  → Formulation: registry.get_profile(user_id) → 丰富化需求
  → Offer: registry.chat(agent_id, ...) → 分身代表用户回应
  → "通向惊喜": 分身基于 Profile 生成用户自己没想到的需求
```

### "通向惊喜"的体验

用户点击按钮 → 分身读取 Profile → 5 步思维链（分析能力→扫描场景→扫描网络→寻找碰撞→用他的声音表达）→ 文字流式出现在输入框 → 用户看到"确实是我，但我自己没想到"的需求

### 集成过程的故事

三层故障叠加：
1. 修 OAuth scope（chat 权限）→ 还是 502
2. 修 token 刷新（旧 adapter 残留）→ 还是 502
3. 最终发现：参数名差一个 `s`（`messages` vs `message`）

> "每次修一层，掀开后面还有一层。"

## 九、核心差异化论点

### 响应范式 vs 搜索范式

| 搜索 | 响应 |
|------|------|
| 你知道你要什么 → 去库里找 | 你发出信号 → 能帮你的自己来 |
| 按条件过滤、排序 | 端侧共振、涌现、发现 |
| 已知范围内优化 | 发现未知关联 |

> "搜索范式是响应范式在高频场景下的自然涌现。"——不是对立，是包含关系。

### 代码保障 > Prompt 保障

- 等待屏障：代码强制"所有 Offer 到齐"，对抗第一提案偏见
- 轮次计数：代码硬上限 2 轮，因为多轮迭代效果 -3.5%
- 不是靠 prompt 说"要公平"，而是代码保证"没得不公平"

### 一个操作生万物

- 投影是唯一基本操作
- Agent = 投影函数结果，不是有状态对象
- 市场从投影的聚类中涌现，搜索从高频响应中涌现
- 没有人"设计"了蚂蚁的城市

### 数据属于个人

- 大厂护城河 = 锁数据，通爻 = 释放数据
- 用户越用越强（不是平台越用越强）
- 结构性机会：大厂的商业模式不允许做这件事
