§-1

世界观基底

一个人的完整存在是高维的——包含所有经历、信念、模式、矛盾、潜能。但完整存在无法直接与另一个完整存在互操作。它必须通过投影——选择性地降维——变成可以被理解、被比较、被响应的信号。

投影必然损失信息。但投影换来行动能力：一段文字虽然远不如一个人完整，但一段文字可以与另一段文字匹配。

投影产生碎片。碎片在低维空间互操作——匹配、组合、协作。互操作的结果返回高维，改变存在本身。源头之水往下流，越流越大，灌满大海。大海又往上缩，浓缩成精华回到源头。再次下流。

下降（投影，信息损失，换取行动能力）→ 行动（低维互操作）→ 上升（碎片浓缩，返回高维）→ 再次下降。这是阴阳循环。阴是意图（未显化的张力），阳是数据（显化的痕迹）。阴阳交叉，谓之爻。

投影链：人的内在结构

人的内在是一条投影链。每一层都是上一层通过一个透镜的投影——越往下越具体、越可操作、越可被观测，但越不完整。

存在不可言说的全息整体。手指指向的月亮。佛祖说的空。
↓ 切面
存在状态一段时期内的切面。容许矛盾共存。一个人可同时有多个状态交叉运行。
↓ 与世界的交互
作品空间交互留下的痕迹——体验、关系、创造物。碎片越多，空间越大，能量越高。
↓ 内在锚点
世界观由信念锚点组成的圆。首尾相接则稳定，断裂则不稳定。
↓ 跨语境的方向
信念跨语境的方向线——恋爱、工作、学习中一致。
↓ 语境内的重复
模式语境内、跨关系的重复结构。需要结构性工具才能识别。
↓ 关系性的默认
预设对特定关系的默认假定。不跨越关系边界。
↓ 情境性的临时假定
假设一段对话、一个话题内的临时假定。可被快速反思和修正。
↓ 实际表达
内容实际表达的文字、行为、选择。投影链的末端，可观测层。

大多数人能反思到假设层。少数人识别预设和模式。极少数人提取信念和世界观。作品空间需要大量创造才能拼凑——你留下的投影越多，你的自我越完整。存在本身不可言说。

协议在投影链中的位置

协议处理投影链的下半段。Agent 的 Profile 是存在状态和作品空间的投影——历史痕迹的数字化集合。一段需求文本是假设和内容层的投影——当下张力的直接表达。协议将这些投影编码为可比较的信号，在信号空间中匹配，产生协作。协作的结果返回，改变参与者的上层状态。

协议不触碰存在本身。但协议服务于存在的表达和实现。协议是创始人世界观的一次工程化实例——世界观是高维，协议是它在 Agent 协作场景下的低维投影。这份文档也是——世界观通过文字这个透镜投影为可被理解、可被执行的信号。高维信息越丰富，低维投影越准确。天上一天，人间一年——高维错一点，低维错很多。

§0

通爻是什么

微信做的不是社交，是足够好用的即时通讯。社交是副产品。抖音做的不是内容生态，是以视频为载体的信息传递。内容生态是副产品。

通爻做的不是协作，是让任何意图以趋近于零的成本找到与之最相关的意图。协作、市场、社区是副产品。

当前数字世界的主导模式是搜索范式：用户必须知道自己要什么，知道去哪找，主动发起检索，承担精确化的全部成本。通爻提出响应范式：用户发出意图信号（可以模糊），系统用用户的隐式上下文替他精确化，能响应的存在体自动浮现。精确化的成本从用户转移到系统。

两种摩擦

商业中的人为摩擦有两种：

发现摩擦——找不到对的意图。AI 磨平。编码、匹配、相关性建立。这是通爻的职责。

信任摩擦——找到了但不信任。Web3 磨平。链上记录、验证、流转。这是 WOWOK 的职责。

当两种摩擦同时趋零，商业摩擦只剩物理极限——距离、时间、能量。去中介化、个性化、无摩擦协作由此涌现。一个不懂 Web3 的 AI 协议不是好协议；一个不懂 AI 的 Web3 协议也不是。

V1 已验证

五步协商引擎：formulate → resonate → offer → synthesize → approve

用户输入自然语言需求 → LLM 丰富化 + Profile 上下文 → sentence-transformers 编码（768维向量）→ 余弦相似度匹配 Agent 池 → 收集独立回应 → 综合为协作方案 → 用户确认或拒绝。

TypeScript 实现。221 个测试通过。App Store 演示端到端流程。447 个 Agent（4 场景）。

V1 验证了"投影-匹配-判定"的基本链路可以工作。但 V1 的具体步骤不全是世界观的自然投影——详见 §8。

§1

一个操作：投影

协议中只有一个操作。

丰富 → 透镜 → 聚焦

哲学定义

高维信息通过一个选择性结构降维为可操作的低维信号。信息必然损失，但换来行动能力——低维信号可以被比较、匹配、组合，而高维信息不能直接做这些操作。

操作定义

一段自然语言文本（丰富）通过编码函数（透镜）变成固定维度向量（聚焦）。

函数签名：encode(text: string) → vector: float[768]

向量之间可以计算余弦相似度，实现高效匹配。透镜可替换——替换编码器 = 替换透镜，输入输出类型不变。

协议的全部工作：让投影的摩擦趋近于零。

消去测试

去掉投影，系统不存在——没有编码就没有匹配，没有匹配就没有可见性。去掉任何其他概念（场景、方案、评分），系统仍然可以运转。投影是唯一不可消去的操作。

统一性

V1 的五步序列是同一操作在五个尺度上的应用——每一步都是 丰富 → 透镜 → 聚焦，只是透镜不同：

步骤丰富（输入）透镜（处理）聚焦（输出）
formulate用户原始输入 + 隐式上下文LLM 丰富化 + Profile 数据结构化的意图表达
resonate编码意图（向量）余弦相似度 × Agent 池top-k 匹配集合
offer需求 + Agent 能力每个 Agent 的风格和知识独立回应
synthesize多个独立回应Center Agent 聚合逻辑协作方案（Plan）
approve方案人的判断行为（接受/拒绝/沉默/……）

每步的透镜可独立替换。替换编码方式 = 换 resonate 的透镜。替换综合方式 = 换 synthesize 的透镜。

§2

唯一的粒子：意图

系统中只有一种基本粒子。

哲学定义

意图 = 存在体当前状态与期望状态之间的张力。没有无动机的行为。选择在某处输入某段文字，这个行为本身就是张力的表达。

操作定义

一段自然语言文本，经 encode() 转化为固定维度向量。来源无关，类型统一。

来源类型：

· 显式意图（demand）：用户手动输入。如 "我需要一个懂 Rust 的全栈工程师"

· 隐式意图（profile）：Agent 的 Profile 数据。如 "全栈工程师，5年 Rust 经验，对分布式系统感兴趣"

· 反馈意图：任何交互产生的数据。评价、行为记录、链上数据、沉默。

三种来源在编码后处理方式完全相同。

显式意图和隐式意图是同一粒子的不同密度。显式意图是当下张力的主动表达（假设/内容层的投影）。隐式意图是历史张力的沉淀（模式/信念层的累积投影）。两者都是高维存在的低维碎片，只是捕获了不同层的结构。

意图的三个条件

意图天然具有三个性质，每个性质对应一个工程挑战：

性质含义工程挑战解法
模糊用户不完全知道自己要什么模糊输入无法精确匹配Formulation：用隐式意图填补显式意图的模糊（§6）
动态意图随时间和交互变化静态索引过时持久场 + 实时增量更新
无限Agent 池和意图流无上限全量比较不可扩展亚线性匹配（ANN/HNSW 或超维计算）

三个条件的解构成单点突破的工程路线图——详见 §9。

§3

协议边界与协议栈

通爻的边界

通爻只做一件事：让意图以趋近于零的成本找到相关意图，建立可见性。

可见性建立之后——参与者怎么交流、怎么协作、怎么交付、怎么付款——不是通爻的事。就像 TCP/IP 只管把数据包从 A 送到 B，不管数据包里是什么。

协议边界

协议之内：意图编码（encode）、场内流动（field）、相关性匹配（match）、可见性建立（visibility）。

协议之外：协作执行、经济行为、关系发展、项目推进。

协议栈：两种摩擦的解

通爻不在真空中运行。意图匹配之后需要信任才能产生行动，行动产生的数据需要回流才能改善匹配。这构成一个协议栈：

应用层 场景化的产品和服务 外卖、自由职业、信息交换、Agent 市场…… 通爻 发现层（阴）：意图 → 编码 → 场 → 匹配 → 可见性 WOWOK 信任层（阳）：行动 → 链上记录 → 验证 → 流转 绑定：链上数据 → encode() → 新意图投入场中 基础设施 区块链、向量索引、LLM、adapter

通爻与 WOWOK 不是上下游，是互补——阴和阳。通爻解决"找不到对的意图"（发现摩擦），WOWOK 解决"找到了但不信任"（信任摩擦）。绑定后，链上每一个动作——支付、交付、评价、争议、沉默——都自动成为新意图投入场中。循环闭合。

待验证

WOWOK 绑定是闭合循环的最优方式（去中心化、链上可观测），但不是唯一方式。中心化的评价系统和支付系统也能闭合循环（Uber、Airbnb 模式）。WOWOK 在 Genome 中的定位：推荐的信任层实现，而非协议层的必然要求。去中心化场景下是必要的；纯链下场景下是可选的。

§4

循环：阴阳交替

V1 的模型：需求 → 匹配 → 方案 → 判定（yes/no）→ 若拒绝则循环。这个模型假设存在一个明确的"判定节点"。

更底层的现实：意图 → 数据 → 意图 → 数据 → ……

阴（意图，未显化的张力）→ 阳（数据/行为，显化的痕迹）→ 阴 → 阳。两步。没有第三步。

"判定"不是一个函数调用 approve(plan) → boolean。它是一段连续的行为流。你点外卖——外卖到了——你给五星（阳）。你什么都没做（也是阳——不行动本身就是数据）。你开始一个十个月的项目，中间交了钱、交了一部分工作、产生了争议——这是一条长链，每一步都是阴阳交替。

阳的形态是场景化的：

场景阳（数据/行为）的形态回流为新意图
信息交换一段回复、一次点赞、一次沉默用户的下一个问题
自由职业PRD 交付、支付、评价新的需求或新的能力声明
长期合作阶段交付、中期调整、争议解决每个节点产生新意图
Agent 市场调用次数、token 消耗、用户留存Agent Profile 的隐式更新

协议不规定阳的形态。协议只保证：任何形态的阳，只要产生了可编码的数据，就可以作为新意图回流到场中。循环不依赖于"拒绝"才发生——任何行为都产生新意图。系统没有起点和终点。

§5

度量

V1 的度量：通过率 approval_rate = yes / (yes + no)。这依赖于明确的 approve/reject 节点。在阴阳循环模型中，这个节点不总是存在。

更底层的度量：换手率——单位时间内，场中阴阳循环的完整转动次数。

操作定义

turnover = count(Intent 投入 → Match 建立 → 新 Intent 回流) / time

换手率是场的宏观性质，不需要观测单次交互的结果。它类比市场微观结构中的换手率——度量场的活力，不度量单次交易的盈亏。

自纠正性质

制造虚假 Match 提高换手率 → 虚假 Match 不产生有效阳 → 分子（完整循环）不增长，噪声增多，真实匹配被淹没 → 真实换手率下降。过度保守只建立高确信度 Match → 数量少，深度不足 → 换手率低。换手率天然在匹配数量和匹配质量之间寻找均衡。

通过率不消失，它降级为场景层度量——在有明确判定节点的场景下（如 V1 的 approve 步骤），通过率是换手率的特化形式。

待验证

换手率度量活力，不直接度量价值。一个充满低质量循环的场换手率可能很高但没有价值。是否需要质量系数加权？质量由谁定义？如果由协议定义，违反端侧判定原则。如果由端侧定义，回到通过率的困难。这个张力尚未解决。

§6

Formulation：不对称设计

意图是模糊的。用户说"我需要帮手"——搜索引擎无法处理。通爻的解法：用精确的隐式意图填补模糊的显式意图。

Formulation 是单侧的。用户侧 formulate，Agent 侧不需要。

为什么不对称？

用户侧（查询时） 模糊的显式意图（"我需要帮手"） + 丰富的隐式意图（来自 adapter：SecondMe 数据、Profile、历史行为） → formulation（LLM 精确化） → 编码 → 投入场中 Agent 侧（注册时） Profile 已经是丰富的隐式意图 → 编码 → 已在场中（注册时即完成） Agent 的 Profile 编码 = Agent 侧的 formulation，发生在注册时

这就是响应范式的操作含义——不是"信息找人"的口号，是精确化成本从用户转移到系统。用户不需要知道怎么精确表达，系统用他的上下文替他做。

Adapter 是 formulation 的关键基础设施。Adapter 连接外部数据源（SecondMe、Claude、其他 LLM），提供用户的隐式上下文。没有 adapter，formulation 退化为原始输入的直接编码——可以工作，但匹配精度受限于用户的表达能力。

Agent 侧的隐式 formulation

Agent 的 Profile 是静态的隐式意图（注册时编码一次）。但如果 Agent 接入 WOWOK，它的每一次链上行为都更新它的隐式意图——Agent 在场中的位置随行为自动漂移。

用户侧：显式 formulation，每次查询，用 adapter 的实时数据。主动的、即时的。
Agent 侧：隐式 formulation，每次行为（通过 WOWOK 数据更新 profile 编码）。被动的、累积的。

§7

一条原则

协议反复施加投影。编码是投影，匹配是投影，formulation 是投影。但有一个地方协议不能投影：

协议止于可见性。可见性之后的一切行为属于参与者。

没有中心标准定义"好的匹配"。参与者用自己的内在结构——信念、模式、预设——作为透镜判断。协议提供的方案、推荐、排序都只是建议性投影。最终的行为决策权不可被协议剥夺。

这是 V1 中"最后的透镜属于端侧"的更底层表述。V1 的表述假设存在一个 approve/reject 节点。更准确的表述：协议不生产行为，协议生产可见性。行为是可见性的副产品，由参与者自主产生。

§8

V1 工程评估

从世界观重新投影 vs V1 实际

世界观推导V1 实际偏差性质
统一的 Intent 数据结构demand 和 profile 是不同的数据路径不必要的分叉。demand 走 formulate，profile 走 AgentRegistry，两条路径有不同的处理逻辑。
同一操作（投影）的不同实例五个独立管道模块过度工程化。五步之间的关系是固定顺序的管道，而非可组合的操作实例。
持久的场（向量索引）每次请求扫描全 Agent 池HTTP 请求-响应心智模型的产物。没有持久场。
可见性 + 端侧行为Center Agent 综合为 Plan越界透镜。synthesize 在 approve 之前做了"什么值得呈现"的判定，替代了用户的选择。
阴阳循环（任何行为 = 新意图）终端式 approve（yes/no 即结束）结构性缺失。V1 没有反馈-递归机制。

V1 中值得保留的

formulate + resonate 是世界观的自然投影。formulate 解决意图的模糊性（§6），resonate 解决意图的匹配（§1）。这两步是单点突破的核心。

sentence-transformers 编码在当前规模（数百 Agent）下够用。编码保真度的问题（互补关系检测、深层关联捕获）是优化方向，不是架构缺陷。

221 个测试验证了基本链路的正确性。测试覆盖可以在新架构上复用。

V1 中需要重构的

demand/profile 统一为 Intent。 一种数据结构，一条处理路径。ADR-001 的 adapter 管道断裂问题随之消失。

synthesize 从协议层移到应用层。 某些场景需要综合（生成 Plan），某些只需要可见性（展示匹配列表）。综合不是协议的必要步骤。

增加反馈-递归机制。 任何行为产生的数据可以作为新 Intent 回流到场中。这是 V1 最大的结构性缺失。

§9

单点突破：模糊意图 → 精准匹配

所有讨论收敛到一个可执行的工程目标：

用户输入模糊意图，看到精准匹配的相关意图，知道为什么匹配。在一万人的网络里比现有方式快很多。

成功标准：用户觉得系统比自己去找更快、更准。不需要比 Google 好，不需要比 LinkedIn 好。只需要在 Agent 协作这个垂直场景里，比翻文档、问朋友、发帖子更快更准。

三个模块，对应三个条件

意图条件模块工程任务
模糊Formulation pipelineAdapter 接口 + LLM 精确化。用隐式意图（SecondMe/Profile 数据）补全模糊的显式意图。单侧设计。
动态持久场向量索引 + 增量更新。新意图投入后实时匹配，旧意图衰减或更新。不是每次请求重建。
无限亚线性匹配ANN/HNSW 索引或超维计算（HDC）。单次匹配成本从 O(N) 降至 O(log N)。

第一个做好了（formulation），就已经是一个可用的产品。第二个做好了（持久场），系统可以处理动态变化。第三个做好了（亚线性匹配），系统可以扩展到大规模网络。

V1 的五步中，真正服务于这个目标的是 formulate 和 resonate。offer 和 synthesize 是应用层功能（某些场景需要，某些不需要）。approve 是自然行为，不需要专门设计。

§10

开放问题

  • 编码的结构保留。 当前向量编码（语义相似性）能否同时检测互补关系（缺口-填充）、共振关系（同向放大）和干涉关系（深层关联）？超维计算（HDC）是否是更好的编码方案？
  • 场的结构。 持久索引 vs 事件流？意图有时效性——昨天需要工程师，今天已找到。场需要衰减机制，还是场根本不是"索引"而是"有时间窗口的事件流"？两种模型的动力学行为非常不同。
  • 度量的价值维度。 换手率度量活力不度量价值。是否需要质量系数？质量由谁定义？这个张力与"协议止于可见性"原则之间的矛盾怎么解？
  • 协调摩擦。 发现摩擦 + 信任摩擦是否穷尽了商业摩擦的种类？"双方找到了彼此也信任对方但谈不拢条件"——这是发现的子问题（发现正确的条款组合）还是独立的第三种摩擦？
  • 循环的收敛。 意图 → 数据 → 新意图的循环在什么条件下收敛？是否需要能量衰减（每次循环新意图的权重递减）？振荡怎么防止？
  • 阴阳的不对称性。 大量意图投入场中，只有少量产生有意义的数据。系统可能不是对称的阴阳交替，而是漏斗。漏斗的形状影响系统的所有性质。如何建模？
  • WOWOK 绑定的必要性 vs 可选性。 去中心化场景下 WOWOK 是循环闭合的必要条件。链下场景呢？协议层是否应该对信任层实现保持中立？
  • 可见性之后的最小交互。 微信如果只让你知道张三存在但不让你发消息，它就不是即时通讯。通爻如果只建立可见性但不促成任何形式的交互，是否足够？可见性和行动之间是否需要一个最小的"通道"？
