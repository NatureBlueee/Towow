## 这个任务在项目中的位置

通爻（ToWow）是一个 AI Agent 协作网络，它的核心理念是"响应范式"——不是像搜索引擎那样你输入条件筛选结果，而是你发出一个信号（"我需要队友"），网络中相关的人自行判断"这跟我有没有关系"，然后主动响应。

**黑客松组队是通爻验证这个理念的第一个真实场景。** 为什么选组队？因为组队天然适合响应范式——你通常不知道最好的队友长什么样（技能匹配只是表面），搜索条件无法捕捉"对味"这种感觉，而且最好的队伍往往来自意想不到的组合。

B1 不写代码。它做的是**场景建模**——深入理解用户在组队过程中的真实行为、心理和痛点，为产品设计提供有据可依的基础。这份建模的价值不会因为技术架构变化而过时。

## 为什么做这件事

Team Matcher 是通爻的第一个完整应用场景。但目前开发是"边做边想"的——先写代码再想用户体验。缺少系统性的场景建模会导致做出来的东西"技术上跑通但用户不买账"。

**用日常语言说**：你不会在没量过窗户尺寸的情况下去买窗帘。场景建模就是"量尺寸"——把用户在组队过程中的真实需求、决策点、痛点搞清楚，然后才能设计出真正好用的产品。

场景建模的价值跨版本有效——用户心理和决策因素不会因为技术架构变化而改变。

## 你要回答什么问题

**核心问题**：黑客松参赛者从"想找队友"到"组成团队"的完整用户旅程是什么？

**子问题**：
- **用户旅程**：参赛者在黑客松前/中/后的行为和心理状态是什么？
- **决策因素**：选队友时，真正在意的是什么？（技能？性格？时区？过往经验？）
- **数据模型**：需要什么信息才能做好匹配？（SecondMe Profile 里哪些维度最重要？）
- **认知转变**：用户以为自己需要 X 类队友，实际需要 Y 类——这种转变在组队中存在吗？
- **响应范式适用性**：组队场景中，"发出信号等共振"比"搜索过滤"好在哪里？是否有反例？

## 我们提供什么

### 概念着陆（通爻术语 → B1 组队场景中的含义）

| 通爻术语 | 在组队场景中意味着 |
|---------|-------------------|
| 投影（Projection） | 参赛者在这次黑客松场景中展现的特定面——同一个人在 AI 黑客松和 Web3 黑客松中的投影不同（强调不同技能和经历） |
| 共振（Resonance） | 队友之间的"对味"——不只是技能匹配，还有节奏感、沟通风格、做事方式。你可能技术很强但"不对味"。共振是端侧判断：每个人自己判断"这个队跟我有没有关系" |
| 响应范式 | "我发出组队信号，让合适的人主动来"vs "我按条件搜索筛选队友"。在组队场景中，响应范式的核心优势是：你可能不知道最好的队友长什么样——搜索条件可能过滤掉最意想不到的好队友 |
| 认知转变 | 参赛者以为需要"一个前端"（要求），实际可能需要的是"一个能快速出原型的人"（需求）——这个人可能是用 No-code 工具的设计师，而非传统前端 |
| 协商（Negotiation） | 组队讨论不是算法匹配后"分配"，而是多个人商量出最好的组合——可能产生"原来我们可以这样分工"的发现 |
| 场景（Scene） | 一次具体的黑客松就是一个场景——有明确的时间边界、主题约束、规模范围 |

### 设计原则（与 B1 的关系）

| # | 原则 | 在组队场景中的意义 |
|---|------|-------------------|
| 0.6 | 需求 ≠ 要求 | 参赛者说"要找前端"是要求，真正需求可能是"能快速验证想法的人"。B1 需要识别这种差距 |
| 0.8 | 投影是基本操作 | 同一个人在不同黑客松中展现的面不同——你的建模需要捕捉这种"场景化投影" |
| 0.7 | 简单规则涌现复杂行为 | 组队规则应该简单（如"发出信号 → 响应 → 聊一聊"），复杂的组合从简单互动中涌现 |
| 0.1 | 最小完整单元 | 组队场景的最小完整单元是什么？至少需要几个人、什么信息才能完成一次有意义的组队？ |

## 子任务分解

### B1.1 — 调研真实黑客松组队行为
**描述**：通过访谈、观察或二手资料（论坛帖子、Twitter 讨论），了解参赛者在组队时的真实行为和痛点。
**依赖**：无
**交付**：调研笔记（至少 5 个信息源）

### B1.2 — 绘制用户旅程地图
**描述**：从"听说有黑客松"到"团队成立并开始协作"，画出完整的用户旅程。标注情绪曲线、决策点、痛点。
**依赖**：B1.1
**交付**：用户旅程图（可以是 Mermaid / Figma / 手绘照片）

### B1.3 — 建立数据模型
**描述**：定义组队匹配需要的数据维度。区分：硬性维度（技能、时区、语言）和软性维度（风格、动机、学习偏好）。分析 SecondMe Profile 中哪些维度最相关。
**依赖**：B1.1 + B1.2
**交付**：数据模型文档（字段定义 + 重要性排序）

### B1.4 — 创建 5 个典型画像
**描述**：设计 5 个代表性的参赛者画像。每个画像包含：背景、动机、表面需求 vs 真实需求、期望的认知转变。
**依赖**：B1.3
**交付**：5 个画像卡片

### B1.5 — 响应范式 vs 搜索范式对比
**描述**：在组队场景中具体分析：用搜索（"按条件筛选"）和用响应（"发出信号等共振"）各自的优劣。如果发现"某些情况下搜索更好"，也是有价值的发现。
**依赖**：B1.2 + B1.4
**交付**：对比分析文档

## 做完了是什么样

### 产出清单

1. **用户旅程地图** — 完整的参赛者组队路径
2. **数据模型文档** — 匹配需要的信息维度
3. **5 个典型画像** — 含认知转变路径
4. **响应范式 vs 搜索范式对比** — 在组队场景中的具体分析

### 三级质量标准

**做完了（基本合格）**：
- 基于真实数据/调研，不是想象
- 包含至少 1 个"意外发现"（之前没想到的）
- 用户旅程中标注了情绪和决策点
- 数据模型区分了"硬性维度"和"软性维度"

**做得好（超出预期）**：
- 调研覆盖了不同类型的黑客松（AI/Web3/社会创新/公司内部）
- 发现了组队场景中"搜索更好"的情况（诚实分析响应范式的局限）
- 画像中的认知转变路径有真实案例支撑
- 数据模型可直接被 Team Matcher 产品使用

**做得出色（产生额外价值）**：
- 提出了一种组队场景的标准建模框架——可被 T1 模板复用到其他场景
- 发现了组队过程中"意外关联"的真实案例——证明响应范式在这个场景下的独特价值
- 产出了一套可以直接用于黑客松现场的"响应式组队工具"设计方案
- 与至少 10 个真实黑客松参赛者做了访谈并有原始数据
