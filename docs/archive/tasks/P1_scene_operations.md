> 创建日期：2026-02-09
> 任务类型：商业运营 × 场景设计
> 优先级：Tier 1（商业落地的前置依赖）
> PRD 状态：已细化
> 依赖：无硬依赖（运营思考不依赖技术实现）
> 关联任务：B1（黑客松场景建模，提供用户侧洞察）、S1（SDK 组队应用，提供技术侧参考）

---

## 这个任务在项目中的位置

通爻的架构设计中（Section 1.4），**场景（Scene）是商业入口**——每个场景（黑客松组队、创业社区、技能交换）就是一个可运营的商业单元。技术已经准备好了（SDK 可用、V1 引擎运行），但"怎么用通爻运营一个场景"这个问题还没有答案。

**P1 是从技术到商业的桥梁**——不是写代码，是写运营方案。目标受众是"想用通爻来运营一个社区/活动/平台的人"。

```
[技术层：SDK + V1 引擎] ← 已就绪
[运营层：怎么用？（你在这里）] ← P1 填补
[商业层：商业模式 + 增长] ← P1 的下游
```

**与 V1/V2 核心开发完全解耦**：P1 的产出是运营文档和策略，不涉及任何代码。

---

## 为什么做这件事

有了锤子不等于知道怎么钉钉子。SDK 和引擎是锤子，但：

1. **冷启动问题**：一个新场景（比如一个新的黑客松组队社区）的 Agent 画像数据从哪来？没有数据，协商就是空转。
2. **用户教育**：用户习惯了搜索范式（"我输入条件 → 系统给结果"）。响应范式（"我发信号 → 别人来响应"）对用户来说是全新的心智模型。怎么让用户接受？
3. **质量保障**：协商结果的质量取决于 Agent Profile 的质量和 LLM 的表现。运营者如何监控和干预？
4. **成功指标**：怎么判断一个场景"运营成功了"？GMV？用户满意度？认知转变率？
5. **规模化**：第一个场景跑通后，怎么复制到下一个场景？

---

## 你要回答什么问题

**核心问题**：一个运营者如何从零开始，用通爻运营一个成功的场景（如黑客松组队社区）？

**子问题**：

1. **冷启动**：场景上线前，需要准备什么？最少需要多少 Agent 数据才能开始？数据从哪来？
2. **用户引导**：如何向用户解释"响应范式"？用什么话术？在什么环节引导？
3. **质量运营**：如何监控协商质量？什么指标表示"协商结果不好"？运营者可以做什么干预？
4. **增长策略**：如何从第一批种子用户增长到稳定规模？网络效应何时开始显现？
5. **商业模型**：运营一个场景的成本结构是什么？可能的收入模式有哪些？
6. **可复制性**：第一个场景的运营经验，哪些可以复制到下一个场景？哪些是场景特定的？

### 通爻的核心概念（运营者需要理解的）

不需要理解技术实现，但需要理解以下概念：

| 概念 | 运营者语言 |
|------|-----------|
| 响应范式 | 用户不是"搜索找人"，是"发出需求等合适的人来"——就像发朋友圈问"谁认识靠谱的装修队？" |
| 投影 | 同一个人在不同场景中展现不同的面——黑客松中你是"全栈开发者"，创业社区中你是"有教育经验的产品经理" |
| 认知转变 | 用户以为需要 X，实际需要 Y——这是通爻最核心的价值，运营者需要设计让这个转变发生的环境 |
| 场景（Scene） | 一个有明确边界的运营单元——有主题、有时间、有人群、有规则 |
| Agent Profile | 用户在场景中的"画像"——技能、经历、偏好。画像质量直接决定协商质量 |
| 共振 | 一种自动匹配机制——相关的人会被自动关联，不需要运营者手动配对 |

## 子任务分解

### P1.1 — 冷启动策略

**描述**：设计一个场景的冷启动方案——从 0 用户到有意义的第一次协商。

**依赖**：无

**交付物**：
- **冷启动 Playbook**（2000-3000 字）：
  - 最小可行数据：一个场景至少需要多少 Agent 画像？画像需要多详细？
  - 数据采集方案：种子数据从哪来？（导入已有社区数据 / 手动创建 / 邀请填写）
  - 首批用户获取：如何找到前 20-50 个愿意试用的人？
  - 首次协商设计：如何确保第一批用户的体验足够好？（运营者可能需要手动补充 Profile 数据）
  - 失败场景预案：如果首次协商结果不好，怎么办？

### P1.2 — 用户引导手册

**描述**：设计用户引导方案——让习惯搜索范式的用户理解并接受响应范式。

**依赖**：无

**交付物**：
- **用户引导手册**（1500-2000 字）：
  - 第一印象：用户打开应用时看到什么？（不是搜索框！）
  - 核心话术：如何用一句话解释"这里不是搜索平台"？
  - 需求表达引导：如何帮用户从"我要一个前端开发者"变成"我想在 48 小时内做出一个可演示的原型"？
  - 认知转变时刻：在什么环节用户最可能发现"原来我需要的不是我以为的"？
  - 常见问题：用户最可能问的 5 个问题及回答
  - 不同用户类型的引导差异（技术用户 / 非技术用户 / 活动组织者）

### P1.3 — 质量运营指南

**描述**：定义运营者监控和提升协商质量的方法。

**依赖**：P1.1

**交付物**：
- **质量运营指南**（1500-2000 字）：
  - 核心质量指标：
    - 协商完成率（需求提交 → 方案产出的转化率）
    - 方案采纳率（方案产出 → 用户实际采纳的转化率）
    - 认知转变率（用户反馈"发现了没想到的东西"的比例）
    - Agent 响应率（被邀请参与协商的 Agent 中，实际产出 Offer 的比例）
  - 质量问题诊断：什么指标异常说明什么问题？
  - 运营干预手段：
    - Profile 补充：帮助用户完善画像
    - Seed 数据注入：在数据不足时手动添加高质量画像
    - Skill 调整：协调 SDK 开发者调整 Prompt（通过反馈而非直接修改）
  - 报告模板：周度/月度运营报告应该包含什么

### P1.4 — 增长策略与商业模型

**描述**：设计场景从种子用户到规模化的增长路径，以及可能的商业模式。

**依赖**：P1.1 + P1.3

**交付物**：
- **增长与商业文档**（2000-3000 字）：
  - 增长阶段划分：
    - 种子期（0-50 人）：手动运营、逐一邀请
    - 验证期（50-200 人）：验证核心价值主张
    - 增长期（200-1000 人）：网络效应开始、口碑传播
    - 规模期（1000+ 人）：自运营、降低人工干预
  - 每个阶段的关键指标和关键动作
  - 网络效应分析：什么条件下"越多人用越好用"开始成立？
  - 成本结构：LLM 调用成本、运营人力成本、基础设施成本
  - 收入模式探索（至少 3 种可能）：
    - 场景方付费（活动组织者购买组队服务）
    - 增值服务（高级 Profile 分析、优先匹配）
    - 数据价值（匿名化的匹配洞察报告）
    - 其他可能
  - 单位经济分析：每次协商的成本 vs 每次协商的收入潜力

### P1.5 — 场景复制框架

**描述**：提炼出从第一个场景复制到下一个场景的标准化流程。

**依赖**：P1.1 + P1.2 + P1.3 + P1.4

**交付物**：
- **场景复制 Checklist**（1000-1500 字）：
  - 新场景评估标准：什么样的场景值得用通爻运营？
  - 标准化流程：从评估 → 冷启动 → 验证 → 增长的步骤
  - 可复用资产：用户引导模板、质量指标体系、运营报告模板
  - 场景特定调整：哪些部分每个场景都要重新做？
  - 案例模板：记录每个场景运营经验的标准格式

## 做完了是什么样

### 产出清单

1. **冷启动 Playbook** — 从 0 到 1 的启动方案
2. **用户引导手册** — 让用户理解响应范式
3. **质量运营指南** — 监控和提升协商质量
4. **增长与商业文档** — 增长路径 + 商业模式
5. **场景复制 Checklist** — 从 1 到 N 的标准化流程

### 三级质量标准

**做完了（基本合格）**：
- 5 个文档全部完成
- 冷启动方案可操作（不是"找到种子用户"这种空话，而是具体的渠道和方法）
- 质量指标有明确定义
- 商业模式有至少 3 种可能

**做得好（超出预期）**：
- 至少 1 个策略经过真实验证（比如在一个小型黑客松中实际使用了冷启动方案）
- 用户引导手册经过真实用户测试（让 5 个人看完后能正确理解响应范式）
- 商业模式有数据支撑的单位经济分析
- 发现了运营中"通爻与传统平台的根本差异"的洞察

**做得出色（产生额外价值）**：
- 运营手册被一个真实的活动组织者用于运营黑客松组队
- 冷启动策略被验证有效，总结出可复用的方法论
- 商业模式分析被投资者认可为可行方向
- 场景复制框架在第二个场景中实际跑通

## 你必须遵守的

### 硬性约束

1. **面向运营者写作**：读者不是程序员，是活动组织者、社区运营者、创业者。不要出现代码、API、数据结构。
2. **可操作性**：每个策略必须具体到"谁做什么、什么时候做、做到什么程度"。不要空泛的"建立社区"。
3. **诚实面对局限**：如果某些运营问题还没有答案，标注为"需要验证"而不是编造答案。
4. **中文优先**：面向中文市场的运营者。

### 与通爻设计原则的对齐

- 运营策略必须体现响应范式——不能退化为"帮用户搜索匹配"
- 冷启动不能依赖"平台拥有所有数据"——通爻的设计是"Profile 数据在数据源"（完备性 ≠ 完全性）
- 用户引导必须帮助用户区分"需求"和"要求"
