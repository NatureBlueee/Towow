> 创建日期：2026-02-07

> 优先级：Tier 1（#4）
> PRD 状态：已细化
> 关联任务：A5（竞品对标，可并行）、D1（响应范式科普，下游）、H3（开发者入门，可并行）

---

## 为什么做这件事

通爻网络的核心概念——投影、共振、响应范式、协商——融合了分布式系统、哲学和原创术语。这套语言对内部团队非常锐利，但对外部合作方几乎不可理解。

当我们和招聘平台说"投影"，对方听到的是"我不知道你在说什么"。当我们和投资人说"响应范式"，对方听到的是"又一个造概念的技术团队"。当我们和产品经理说"完备性不等于完全性"，对方听到的是"太学术了"。

这不是因为对方理解力不够，而是因为**我们没有做投影**。

"概念翻译"本身就是"投影"的应用：同一个核心概念，通过不同的"领域透镜"，产生对方能接收的表达。我们用一套语言描述了世界，但世界的不同部分有自己的语言。翻译不是简化——是找到对方领域中的**等价概念**。做不好这一步，通爻的所有外部合作、社区建设、融资沟通都会卡住。

---

## 你要回答什么问题

### 核心问题

**如何让不同领域的人在 3 分钟内理解通爻的核心价值，并能用自己领域的语言向别人转述？**

### 子问题

1. 通爻最核心的概念有哪些？（从 10+ 个候选中筛选出最小必要集合——向外沟通时必须传达的概念）
2. 每个核心概念在目标领域中是否存在等价概念？如果不存在，最接近的类比是什么？
3. 等价映射的边界在哪？（哪些地方"像但不是"，需要明确标注差异）
4. 每个领域的人最关心什么？（他们的"张力"是什么，通爻如何触及？）
5. 哪些场景明确**不适合**通爻？（反例同样重要——告诉别人"我不做什么"是建立信任的最快方式）
6. 翻译后的表达在真实对话中是否 work？（不能只是文档上好看，要在 3 分钟口头对话中验证）

| # | 原则 | 与概念翻译的关系 |
|---|------|-----------------|
| 0.5 | 需求 ≠ 要求 | 翻译时不要按对方的"要求"去扭曲概念，找到对方真正需要理解的东西 |

## 子任务分解

### H1.1 — 核心概念清单定稿

**描述**：从架构文档和设计日志中提取通爻对外沟通时必须传达的核心概念，确定最小必要集合（10-15 个），并为每个概念写一段 50 字以内的"一句话本质"。

**依赖**：无（可立即开始）

**交付物**：
- 核心概念清单（10-15 个），按重要性排序
- 每个概念的"一句话本质"（50 字以内，用于快速理解）
- 概念之间的关系图（哪些是基础概念，哪些是派生概念）

**建议概念候选池**（从文档中提取，不限于此）：
1. **投影（Projection）** — 丰富的东西通过透镜变成聚焦的东西
2. **共振（Resonance）** — 信号到达后，相关的存在自行判断并响应
3. **响应范式（Response Paradigm）** — 不去搜索，而是发出信号等待响应
4. **搜索范式（Search Paradigm）** — 作为对比物存在
5. **协商（Negotiation）** — 多方响应聚合为超越任何单方的方案
6. **透镜（Lens）** — 决定投影方向和聚焦度的参数
7. **完备性（Completeness）** — 与信息保持连通，而非复制信息
8. **回声（Echo）** — 结果反馈回系统，让系统从机器变成有机体
9. **面具 / Service Agent** — 同一个人在不同场景下的不同投影
10. **"自"与"我"** — 人的完整存在（系统外）vs 投影（系统内）
11. **缺口递归（Gap Recursion）** — 方案不完整时触发子需求，分形生长
12. **签名（Signature）** — 信息的振动特征，用于快速判断相关性
13. **意外度（Surprise）** — 区分标准需求和涌现机会的度量
14. **需求 vs 要求** — 真正需要的 vs 以为怎么满足
15. **涌现（Emergence）** — 简单规则递归产生的非预设复杂性

**判断标准**：如果拿掉某个概念，无法向外界解释通爻"是什么"或"为什么不同"，那它就属于最小必要集合。

### H1.2 — 目标领域选定与对话场景设计

**描述**：选定 5-7 个目标领域，为每个领域设计一个**真实的对话场景**——"在什么情况下，我会需要向这个领域的人解释通爻？对方最可能问什么问题？"

**依赖**：H1.1（需要知道翻译什么概念）

**交付物**：
- 目标领域清单（5-7 个），每个领域附带：
  - 领域描述（一句话，这个领域的人关心什么）
  - 典型对话场景（什么场合、谁对谁说、对方的期待是什么）
  - 对方最可能问的 3 个问题
  - 对方领域中的"张力"（什么问题困扰他们，通爻可能触及）

**建议目标领域**（不限于此，可增减）：

| 领域 | 典型场景举例 | 他们的张力 |
|------|------------|-----------|
| **招聘 / HR** | 向 HR Tech 公司介绍通爻的合作价值 | 精准匹配 vs 长尾人才发现 |
| **区块链 / Web3** | 在 Web3 黑客松中向开发者解释通爻的协议层 | 去中心化身份 + 服务发现 |
| **产品管理** | 向 PM 解释"为什么通爻不是一个推荐系统" | 用户需求理解 + 方案生成 |
| **投资 / VC** | 5 分钟内向投资人讲清楚通爻的商业故事 | 网络效应 + 差异化 + 壁垒 |
| **学术研究** | 在 AI Agent 学术会议上定位通爻的研究贡献 | 多智能体协作 + 分布式系统 + 涌现 |
| **自由职业 / 创作者经济** | 向 freelancer 平台解释"响应范式"的价值 | 被动获客 + 超额满足 |
| **教育 / 社区** | 向教育平台解释"每个学生都是不同投影" | 个性化 + 不是标签化 |

### H1.3 — 逐领域概念映射

**描述**：对 H1.1 确定的每个核心概念，在 H1.2 选定的每个目标领域中找到等价概念或最佳类比。这是本任务的核心产出。

**依赖**：H1.1 + H1.2

**交付物**：概念映射矩阵（核心产出），格式如下：

| 通爻概念 | 本质（不变的） | 招聘/HR | 区块链/Web3 | 产品管理 | 投资/VC | 学术研究 | ... |
|---------|-------------|---------|------------|---------|--------|---------|-----|
| 投影 | 丰富→透镜→聚焦 | ... | ... | ... | ... | ... | ... |
| 共振 | 端侧自主判断相关性 | ... | ... | ... | ... | ... | ... |
| ... | ... | ... | ... | ... | ... | ... | ... |

**每个映射单元需要包含**：
- **等价概念**：对方领域中最接近的概念名称
- **一句话翻译**：用对方的语言说的一句话
- **像但不是**（边界）：这个翻译在哪里会误导，需要注意什么
- **对话用法**：在真实对话中怎么说（口语化）

**示例**（展示期望的深度）：

> **概念**：响应范式
> **领域**：招聘/HR
> **等价概念**：被动招聘（Passive Recruiting）
> **一句话翻译**：不是你去搜简历，而是职位发出信号后，匹配的人才自己判断并主动响应
> **像但不是**：被动招聘依赖猎头推荐（中心化），通爻的响应是端侧自主判断（去中心化）；被动招聘只能触达"被标记为开放"的人，通爻能触达不知道自己适合这个岗位的人
> **对话用法**："你知道被动招聘吧？通爻做的事情类似，但更彻底——不需要猎头，候选人的AI代理自己判断'这个机会跟我有关'，包括候选人自己都没想到的方向。"

### H1.4 — 每个领域的"一页纸"

**描述**：基于概念映射，为每个目标领域撰写一页纸（500-800 字），标题是"通爻能为 [领域] 做什么"。

**依赖**：H1.3

**交付物**：
- 每个目标领域一份"一页纸"（5-7 份）
- 统一结构：
  1. **你面临的问题**（用对方领域的语言，40-80 字）
  2. **当前的解法和局限**（搜索范式在这个领域的表现，80-120 字）
  3. **通爻提供的不同视角**（用翻译后的概念，150-250 字）
  4. **一个具体场景**（对方能想象自己身处其中的故事，100-200 字）
  5. **这不是什么**（明确的边界声明，50-100 字）

**质量标准**：给目标领域的一个从业者看，他/她能在 3 分钟内读完、理解核心价值、并能用自己的话向同事转述一句话版本。

### H1.5 — "不适合通爻"反例清单

**描述**：整理通爻**不适合**的场景清单。这不是示弱——这是精准定位的另一面。当你能清晰地说"我不做什么"，别人才相信你说的"我做什么"。

**依赖**：H1.3（在映射过程中会自然发现不适合的场景）

**交付物**：
- 反例清单（8-12 个场景），每个场景包含：
  - 场景描述（一句话）
  - 为什么不适合（核心原因）
  - 适合用什么（替代方案）

**建议方向**（不限于此）：
- 需要精确匹配且无容错空间的场景（如：紧急医疗资源调配）
- 需求完全明确、不需要发现的场景（如：买一瓶特定品牌的水）
- 参与者太少、无法产生协商价值的场景
- 纯粹信息检索场景（搜索范式已经足够好）
- 实时性要求极高、无法等待协商的场景
- 隐私敏感度极高、不适合投影的场景

### H1.6 — 使用指南与对话手册

**描述**：编写实际使用指南——怎么在真实对话中运用这些翻译。不是学术文档，而是"话术手册"。

**依赖**：H1.4 + H1.5

**交付物**：
- **对话手册**（2000-3000 字），包含：
  1. **通用开场**：不分领域的 30 秒电梯演讲（3 个版本：技术人、商务人、普通人）
  2. **领域切入**：每个领域的"第一句话怎么说"
  3. **常见误解对照**：对方最容易误解的 5 件事，以及怎么纠正
  4. **FAQ 速查**：10 个最常被问到的问题和推荐回答
  5. **升级路径**：如果对方感兴趣，下一步推荐阅读什么 / 体验什么

## 做完了是什么样

### 核心产出（6 份文档）

1. **核心概念清单**：10-15 个概念，每个有一句话本质 + 关系图
2. **目标领域清单**：5-7 个领域，每个有对话场景和 3 个关键问题
3. **概念映射矩阵**：核心概念 x 目标领域 的完整映射表，每个单元含等价概念、翻译、边界、对话用法
4. **领域一页纸**：5-7 份"通爻能为 [领域] 做什么"
5. **反例清单**：8-12 个"不适合通爻"的场景
6. **对话手册**：真实场景中的使用指南

### 总规模

- 概念映射矩阵：约 3000-5000 字（核心知识资产）
- 领域一页纸：每份 500-800 字 x 5-7 份 = 2500-5600 字
- 对话手册：2000-3000 字
- 总计约 10,000-15,000 字

### 质量判断标准

| 层级 | 标准 |
|------|------|
| **做完了** | 所有 6 份文档齐全，格式完整，概念无硬伤 |
| **做得好** | 拿给目标领域的从业者看，对方在 3 分钟内理解核心价值，能用自己的话转述 |
| **做得出色** | 翻译中发现了新的等价概念，反过来加深了我们对自己概念的理解（投影的逆操作——多个聚焦的翻译汇聚，让原始概念更清晰） |

## 你必须遵守的

### 1. 翻译是投影，不是简化

不要把"投影"翻译成"用户画像"——那不是投影，那是画像（完全性思维）。投影的关键在于"有损但保留本质"。如果你的翻译丢掉了概念的**本质区别**（通爻与传统方案的核心差异），那就不是翻译，是稀释。

**测试方法**：用你的翻译替换原概念，通爻是否还与竞品有差异？如果没有差异了，翻译有问题。

### 2. 每个翻译必须标注边界

没有完美的翻译。每个等价概念都有"像但不是"的部分。**必须显式标注**翻译的有效边界和可能的误导方向。

### 3. 要有真实对话场景支撑

不能只是文档上的表格。每个翻译必须通过"我在 3 分钟对话中能说出来"的测试。如果一个翻译写在纸上好看但说不出口，重做。

### 4. SecondMe 不是唯一的 Profile 数据源

在提到通爻的数据来源时，不要写成"用户通过 SecondMe 接入"。通爻使用 Adapter 模式（见 Design Log #003），SecondMe 只是其中一个适配器。正确的表达是"通爻从各种数据源（如 SecondMe、Claude、GPT 等）获取投影数据"。

### 5. 对齐核心设计原则

所有翻译必须与以下原则保持一致（不能翻译出矛盾的含义）：
- 需求 ≠ 要求（不要把通爻翻译成"精准匹配系统"）
- 投影是基本操作（不要把概念翻译成暗示"完全理解用户"的表达）
- 搜索是响应的涌现（不要把搜索和响应对立为互斥关系）
- 一自多我（不要暗示一人一Agent的固定映射）

### 6. 中文为主

所有产出文档使用中文。概念的英文名称在首次出现时括注。

## 你可以自己决定的

### 1. 目标领域的选择

建议了 7 个领域，但你可以增减。如果你认为某个领域更有价值（比如你有该领域的从业经验），可以替换。最终不少于 5 个。

### 2. 概念清单的取舍

候选池给了 15 个概念，最终清单由你根据"对外沟通的最小必要"来筛选。可以增加我们没列出的概念，也可以合并或拆分。

### 3. 类比和比喻的选择

已有文章中用了"窗户 vs 照片"、"全息碎片"、"蝙蝠回声定位"等类比。你完全可以发明新的、更好的类比。好的类比 = 目标领域的人能在 3 秒内联想到自己的经验。

### 4. 产出格式的细节

一页纸的结构、对话手册的分类方式、表格 vs 叙述——这些格式细节由你决定，只要最终产出是实用的。

### 5. 深度 vs 广度的取舍

如果某个领域的映射特别有深度（你恰好有该领域专长），可以在该领域做得更深，其他领域保持基本覆盖。**不需要每个领域都一样深**——这本身就是"投影"的实践。

### 6. 额外发现

如果在翻译过程中发现：
- 通爻的某个概念定义不够清晰（翻译时才暴露的模糊地带）
- 某个领域的等价概念反过来能启发通爻的设计
- 某个翻译意外地比原始概念更易传播

请记录下来。这些发现本身就是宝贵的产出。
