## 1. 我们要什么

让真人的需求通过催化过程被真正消解。

不是"证明催化能工作"，是"找到什么因素让催化工作得更好"。成功标准：需求方看到最终产出后说"这真的能帮到我"，而不只是"看起来挺有意思"。

---

## 2. 与 AI 模拟的本质区别

| 维度 | AI 模拟 (SIM-001/002) | 真人实验 |
|------|----------------------|---------|
| Profile | LLM 生成，结构工整 | 真人写的，格式各异，深浅不一 |
| 语义鸿沟 | 同一模型制造的假差异 | 真实的领域差异、表达习惯差异 |
| 需求 | 预设场景（CEO 转型） | 真实需求，有的模糊有的具体 |
| 评估 | 预埋 Ground Truth + LLM 评分 | **需求方本人判断** |
| 信号质量 | 模糊（系统在自言自语） | 清晰（输入和评估都是真的） |

## 3. 从 AI 实验中学到的（经验观察，不是规则）

以下每条标注了"在什么条件下发现的"，在真人数据上可能表现不同。

| # | 观察 | 发现条件 | 真人实验中可能... |
|---|------|---------|-----------------|
| 1 | 催化翻译在 R3 后退化为同域归纳 | 3 人 × 6 轮，合成 Profile | 真实语义鸿沟更深，退化可能更晚或不退化 |
| 2 | 翻译固定 4 条/轮（模式锚定） | catalyst_v1 的格式要求 | 同一 prompt 应该还会出现，需要 v2 改 |
| 3 | max_tokens=3000 截断收敛判断 | run_sim.py 硬编码 | 纯工程问题，提到 6000+ 即可 |
| 4 | 催化不区分真实信息差和讨论副产品 | v1 prompt 没给操作标准 | 真人数据上更关键——真实讨论会产生更多副产品 |
| 5 | 端侧太被动，催化填补空白 | endpoint_v0 没有 R2+ 指引 | 真人 Profile 信息更丰富，端侧可能更主动 |
| 6 | 方案生成器把单方提议当共识 | plan_generator_v0 的问题 | 独立于数据，prompt 级别问题 |
| 7 | 五级优先级未被催化执行 | v1 列出了优先级但未操作化 | 需要 v2 把优先级变成操作步骤 |

**使用原则**：这些是线索，不是定论。真人数据上发现新问题时，新问题优先级更高。

### 4.3 Profile 预处理

真人 Profile 格式各异。预处理原则：
- **不改写内容**：保留原始表达，包括模糊的、口语化的、不完整的
- **只做结构标注**：在不改变内容的前提下，标注"这段在说能力""这段在说需求""这段在说约束"
- **记录丰富度**：Profile 信息量差异本身是实验变量之一

## 5. 实验单元设计

### 一次实验 = 一个"结晶 session"

```
输入:
  - 1 个需求（触发事件）
  - 3-5 个 Profile（参与者，从池中选取）
  - 催化 prompt 版本（从 v1 开始）
  - 端侧 prompt 版本（从 v0 开始）

过程:
  - 每轮：端侧 Agent 基于 Profile 回复 → 催化观察 → 注入下一轮
  - 最大轮次：6（可调）
  - max_tokens：6000+（修复已知截断问题）

输出:
  - 每轮原始对话记录
  - 最终方案（plan）
  - 实验元数据（模型、参数、成本、时间）
```

### 参与者选择策略

第一个实验：**不刻意挑选**。从池中选一个真实需求，然后看池中谁可能相关就拉进来。不预设 Ground Truth——真人数据的价值就在于我们不知道会发现什么。

后续实验：可以刻意设计对照（比如同一需求换不同参与者、同一参与者换不同需求）。

## 6. 评估方法

### 6.1 先理解需求方

每个实验开始前，先深入分析需求方：
- 读完整 Profile
- 理解 TA 到底想要什么（不是表面说的，是真正的张力）
- 写下"如果完美消解，TA 会得到什么"
- **用户确认**这个理解是否准确

### 6.2 过程评估（用户逐轮判断）

每轮展示原始输出，用户回答：
- 催化这轮有没有帮到需求方？
- 有没有做了不该做的事？
- 端侧回复质量是否在进步？

### 6.3 终态评估（核心）

最终方案展示给用户，用户回答：
- **这个方案消解了需求方的原始张力吗？**
- 需求方看到这个会说"这真的帮到我了"还是"嗯，挺有意思的"？
- 方案中哪些是惊喜（超出预期的价值），哪些是噎塞（不该出现的东西）？

### 6.4 评估记录格式

```markdown
## RUN-{NNN} 评估

### 需求方理解
[TA 到底想要什么]

### 逐轮判断
- R1: [用户判断]
- R2: ...
- ...

### 终态判断
[方案是否消解张力？惊喜？噎塞？]

### 识别的因素
[什么在影响效果？下一次改什么？]
```

## 7. 迭代机制

### 迭代循环

```
RUN-N
  ↓
用户评估：什么因素在影响效果？
  ↓
识别最关键的 1 个因素
  ↓
修改对应的 prompt（催化 or 端侧）
  ↓
RUN-N+1（同一场景重跑，或换场景验证泛化性）
  ↓
用户评估：这个因素改了之后效果变了吗？
  ↓
重复
```

### 迭代记录

每次迭代产出：
1. **因素识别**：这次发现什么在影响效果
2. **修改内容**：prompt 的哪一段改了什么
3. **修改理由**：为什么认为这样改会更好
4. **验证结果**：改了之后效果如何（下一次 RUN 后填）

### 迭代终止条件

- 用户判断"这个效果可以了"→ 进入更多场景验证泛化性
- 或者发现是设计层面的问题（不是 prompt 能解决的）→ 回到 DESIGN_LOG_006 讨论

## 8. 运行基础设施

### 脚本

基于 `tests/crystallization_poc/simulations/sim002_3person_v1/run_sim.py` 改造：
- 支持任意数量参与者（3-8）
- 支持不同格式的 Profile 输入
- max_tokens 参数化（不硬编码）
- 输出目录：`tests/crystallization_poc/simulations/real/run_{NNN}/`

### 成本估算

- 每轮 ≈ (N 个端侧调用 + 1 个催化调用) × token 数
- 3 人 × 6 轮 ≈ ~$2-4（Sonnet）/ ~$8-15（Opus）
- 建议：催化用最好的模型（Opus），端侧用 Sonnet

### 文件结构

```
tests/crystallization_poc/simulations/real/
├── run_001/
│   ├── config.json          # 实验配置（参与者、prompt 版本、参数）
│   ├── trigger.md           # 需求描述
│   ├── participants.md      # 参与者清单
│   ├── output/
│   │   ├── round_1.md ... round_N.md
│   │   ├── plan.md
│   │   ├── transcript.md
│   │   └── metadata.json
│   └── evaluation.md        # 用户评估记录
├── run_002/
│   └── ...
└── iteration_log.md         # 跨 RUN 的迭代记录
```
