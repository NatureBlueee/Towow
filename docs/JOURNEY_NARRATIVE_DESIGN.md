## 二、七个阶段的深度内容

### 阶段 1：探索与理解（1/21 – 1/22）

**核心叙事**（136字）
一切从理解 OpenAgents 框架开始。这不是一个"新项目启动"的故事，而是一个"站在巨人肩膀上找路"的过程。我们发现 ToWow 的生产环境偏离了 OpenAgents 的设计范式——自研的路由和 Mock 通道只打日志不通信。而 Raphael 演示项目虽然简单，却完整跑通了多 Agent 协作。修复了 Python 导入问题后，第一次看到 Agent 们自主协调工作，那一刻确认了方向：不要重新发明轮子，要基于已验证的模式来扩展。

**关键决策**
- 选择以 Raphael 演示代码为基础，而非修复偏离框架的生产代码
- 使用绝对导入 + PYTHONPATH 解决模块加载问题（工程上不优雅，但有效）
- 确定"每个用户都是一个 Worker Agent"的方向

**用户纠正**
无明显纠正。这个阶段是"学习和对齐"，创始人在引导方向。

**意外发现**
ToWow 生产环境的 `_MockChannelHandle` 只打日志不通信——这不是 bug，而是架构选择错误的信号。偏离了成熟框架的代价比想象中大。

**关键引用**
> "这个非常棒！！！" ——创始人看到多 Agent 协作跑通后的反应

### 阶段 2：后端服务封装（1/27 – 1/29）

**核心叙事**（142字）
有了能跑的骨架，下一步是让它可用。SecondMe OAuth2 集成让用户能用真实身份登录；SQLite 数据层让注册信息能持久化；WebSocket 让状态变化能实时推送。这个阶段最重要的不是技术实现，而是创始人传递的设计理念——"Keep it simple"。当我们的技术方案过度设计（复杂的并发控制、细粒度的限流），创始人直接打断："过度的复杂了"、"自我揣测了"。服务器够强，用户够少，先让它能用。

**关键决策**
- 使用 SQLite 而非 PostgreSQL（足够轻量，足以验证）
- OAuth2 数据单向流动：SecondMe → ToWow（不把用户在 ToWow 的数据推回 SecondMe）
- 技术方案从"生产级"简化为"验证级"

**用户纠正**
- "过度的复杂了" —— 拒绝了包含速率限制、并发控制的复杂方案
- "自我揣测了" —— 不要替用户假设他们的需求和限制
- "Keep it simple" —— 最小可用比完美重要

**意外发现**
SecondMe API 不返回 `openId` 字段——与文档不符。用 `email` 作为替代标识符，绕过了这个问题。真实系统总有文档没说的意外。

**关键引用**
> "不需要质疑那么多" ——创始人对过度审慎的 Agent 的回应

### 阶段 3：网站全栈开发（1/30）

**核心叙事**（168字）
这是整个历程中最密集的一天——20次提交，8次上下文压缩，从一个空白的 Next.js 项目到一个可部署的完整网站。首页6个板块的文字从"200字一屏"精简到"100字以内"；配色从冷灰调整到暖米白；布局从网格卡片改为流动渐变。最有意义的是 Demo 演示场景的设计：创始人否决了"独立音乐人打造演出"的方案，因为它不体现通爻的核心价值。改为"找技术合伙人"——用户以为需要一个人，协商后发现真正需要的是一种能力。这个演示场景本身就是通爻哲学的微缩版。

**关键决策**
- Next.js 16 + App Router + CSS Modules（不用 Tailwind，保留对样式的完全控制）
- 移除网格/卡片设计，改用滚动渐变背景（"网格给人更多囚禁的秩序感"）
- Demo V2 的10阶段交互动画：从需求缩小为点到最终方案形成

**用户纠正**
- "网站太冷/临床感" —— 从 #EEEEEE 冷灰到 #F8F6F3 暖米白，不仅是颜色，是气质
- "这个独立音乐人打造演出是完全没有用的东西" —— 演示必须体现哲学，不是随便找个场景
- "视角偏右...左边有大概百分之二十的部分看不到" —— 提醒关注细节

**意外发现**
React Strict Mode 导致 WebSocket 双挂载/卸载——组件挂载两次，第一次的 WebSocket 被第二次卸载清理掉。生产环境不会有这个问题，但开发时让人误以为 WebSocket 坏了。

**关键引用**
> "你以为你要一个什么东西，结果发现你不需要。" ——创始人定义"认知转变"

### 阶段 4：部署与运维（1/31）

**核心叙事**（119字）
网站从本地跑通到线上可访问——Railway 后端 + Vercel 前端 + Cloudflare CDN。这是一个看起来"只是部署"但实际充满细节的阶段：OAuth 回调需要支持多个地址（本地和生产）、Cookie 的 SameSite 策略影响跨域认证、Vercel 的 Root Directory 要指向 monorepo 的子目录。每一个"小问题"都需要理解整个请求链路。部署不是终点，而是让所有人能看到的起点。

**关键决策**
- 支持 `USE_REAL_AGENTS` 环境变量切换真实/模拟模式
- Cookie 安全策略通过环境变量控制（`COOKIE_SECURE`）
- 保留 Experience V2 演示版本，同时创建 V3 真实 Agent 版本

**用户纠正**
无明显纠正。这个阶段是工程执行，方向已明确。

**意外发现**
Vercel 在 monorepo 中找不到 Next.js 目录——需要在 Dashboard 设置 Root Directory 为 `website/`。部署配置比代码更容易被忽视。

**关键引用**
（本阶段以工程执行为主，无突出的创始人引用）

### 阶段 5：架构深度重构（2/4 – 2/6）

**核心叙事**（196字）
这是整个历程的分水岭。创始人说了一句话："设计得很理想，但架构就是出很多小错误。" 然后他做了一个出乎意料的决定——不是让我去修 bug，而是让我停下来读文档。三篇核心文档：白皮书、技术简报、设计日志。读完之后不是写代码，而是讨论。接下来3天，14次上下文压缩，全部用于架构思考。从签名广播的 HDC 超向量编码，到 Agent 接入的多源适配器，到 Skill 系统的"提案-聚合"范式，到 WOWOK 区块链协议的集成——每一轮讨论都是对"本质是什么"的追问。最后一轮讨论 Service Agent 结晶机制时，创始人说"本质和实现没有区分开来，不应该这么复杂"。然后，在多次思维跳跃后，"投影即函数"诞生了——Agent 不是对象，而是函数结果；投影是唯一的基本操作。

**关键决策**
- 三层共振过滤：Bloom Filter（O(1)）→ HDC 共振（O(D)）→ LLM（O(model)）
- "代码保障 > Prompt 保障"：凡是能用代码保障的确定性逻辑，绝不用 prompt
- 多轮辩论是净负面（-3.5%，DeepMind 2025），但并行提案→聚合是正面（+57-81%）
- WOWOK 区块链作为"回声"机制——波浪出去了，必须回来
- "投影即函数"：消除了 Profile 更新算法、防漂移机制、状态维护、冷启动问题

**用户纠正**
- "不要为了做MVP而做MVP，不要为了单纯的削减功能而用削减功能" —— 最小完整单元 ≠ 砍功能
- "你只是给我执行了，但你没有想清楚" —— 不要先写再想，要先想再写
- "先讨论确认理解，再写文档" —— 对齐 > 效率
- "本质和实现没有区分开来" —— 催生了"投影即函数"
- "有些问题本身就不该出现" —— 最好的解决方案是消除问题本身
- "SecondMe 不仅仅是在 SecondMe，它也在其他地方应该能做到这个事儿" —— 可插拔 > 绑定

**意外发现**
- 研究发现 LLM 的"第一提案偏见"高达 10-30 倍（Microsoft 2025）——这不是 prompt 能解决的，必须用代码（等待屏障）保障
- WOWOK 协议的 Guard 不是"签名"而是"验证引擎"——理解错误导致回声机制的设计基于错误前提
- "广播和筛选是同一个逻辑执行了两次"——同一个数学操作，反转方向

**关键引用**
> "设计得很理想，但架构就是出很多小错误。" ——触发转变的那句话
> "应该要极度简单。" ——创始人对"投影即函数"方向的确认

### 阶段 6：黑客松工具链（2/7）

**核心叙事**（138字）
架构想清楚之后，回到工程——但这次不一样了。每个工具都先用 arch skill 的方法论思考"做什么"和"为什么做"，然后才开始实现。Team Matcher 不是"组队匹配"，而是"协作可能性发现引擎"；Guide 不是"教程"，而是"认知透镜"。最戏剧化的时刻是应用卡在"广播中..."无反应——测试通过但实际不工作，mock 数据掩盖了真实的集成失败。创始人爆发了："mock模式不重要，直接变成应用！" 这句话与架构讨论中的"不要模拟"一脉相承。

**关键决策**
- 使用 Opus 4.6 并行 Agent 开发（最多9个同时工作）
- SecondMe Chat API 集成：登录后 AI 基于 Profile 自动建议表单值
- 放弃 mock 模式优先的策略，直接面对真实集成的复杂性

**用户纠正**
- "mock模式不重要，直接变成应用" —— 不要为了演示而演示
- "不要说什么模拟五个Agent去响应" —— 同上
- 明确要求用 Opus 4.6 而非 Sonnet 4.5 —— 质量 > 速度

**意外发现**
并行 Agent 开发的核心问题不是代码质量，而是接口对齐。4个 Agent 独立开发，产生了5个集成 Bug（WebSocket 消息被丢弃、Channel ID 不匹配、字段名不一致等）。需要一个"跨边界审查者"角色。

**关键引用**
> "mock模式不重要，我需要全部的登陆啥的，直接做能用的，不是为了mock而mock" ——创始人

### 阶段 7：共建任务体系（2/8）

**核心叙事**（156字）
19天的最后一天，把所有成果打开给社区。33个初始任务用架构评估精简为18个——"不是为了有任务而有任务"。9个 Opus 4.6 Agent 并行编写全部18篇 PRD，每个 Agent 使用 arch + task-arch skill 确保哲学对齐。/contribute 页面经历了三次迭代：第一版太花哨，第二版太简洁，创始人说"不是为了简洁而简洁"——每一层信息都有存在的理由，移除任何一层会丢失意义。这个设计原则恰好就是通爻自身的 Section 0.9（完备性 ≠ 完全性）的具体化。一次提交，165个文件，+18,795行。

**关键决策**
- 33→18 任务精简：用架构原则（"这个任务的本质是什么？它解决的问题是真实的吗？"）筛选
- 飞书为主 + 网站为辅的共建策略（Bitable 管理 + build in public）
- PRD 写作使用 arch skill 方法论，确保"技术细节的背后有世界观"

**用户纠正**
- "不是为了有任务而有任务" —— 33→18 不是因为数量太多，而是价值不够
- "不是为了简洁而简洁" —— 极度简单但保留完整意义（三次页面迭代的核心教训）
- 纠正了 AI 对架构设计承诺的误解 —— "你误解了设计承诺的含义"

**意外发现**
9个 Agent 并行写 PRD 时，每个 Agent 对"通爻哲学"的理解略有不同——这本身就是"投影"的一个实例：同一个架构文档，通过不同的 Agent "透镜"，产生不同但各有价值的 PRD。

**关键引用**
> "不是为了简洁而简洁。" ——与 Section 0.9 完美对应的设计原则

## 三、41个 Compact 的展开摘要

### 会话 1：主开发会话（1/26 – 2/5）

**Compact #1** · 1/26 · 理解项目 + 修复 OpenAgents 导入问题 + 成功跑通多 Agent 协作
第一次深入 ToWow 代码。对比发现 Raphael 演示用原生 OpenAgents 模式（事件驱动、BaseMod），而生产环境偏离了框架（自研路由、Mock 通道只打日志）。Python 相对导入问题阻止了 Mod 加载，用 `sys.path` 操作绕过。修复后完整跑通了需求提交→频道创建→Agent邀请→任务分发→协调汇总的流程。确认了以演示代码为基础扩展的方向，开始设计动态用户注册系统。

**Compact #2** · 1/27 · SecondMe OAuth2 登录 + 动态 Agent 创建
创建 Web 注册服务，接入 SecondMe OAuth2 授权码流程。用户登录后自动创建 Worker Agent 并注册到网络。完整的安全审查（Tech Agent）发现8个问题，全部修复。17个单元测试通过。数据流是单向的：SecondMe → ToWow，用户在 ToWow 的新信息不回传。

**Compact #3** · 1/27 · GitHub 上传 + OAuth 测试 + URL 编码问题修复
配置 `.gitignore` 排除大文件，推送到 GitHub。测试 SecondMe OAuth 流程时发现回调地址被双重编码（`urlencode` 编码后又被框架编码），移除手动编码修复。发现 SecondMe API 不返回文档中的 `openId` 字段，改用 `email` 做标识符。

**Compact #4** · 1/29 · 后端生产化封装 — SQLite + WebSocket
创始人要求支持 2000-3000+ 并发，但同时强调"Keep it simple"。技术方案第一版过度设计（复杂并发控制、限流），被否决。简化为 SQLite + SQLAlchemy 数据持久化、WebSocket 实时推送。16个 API 测试通过。

**Compact #5** · 1/29 · 测试 + API 文档 + 前端 HTML → Next.js 迁移
完整 API 测试跑通，生成 API 文档。开始将 HTML 首页和文章详情页迁移到 Next.js 14（App Router），采用 CSS Modules + CSS Variables。技术方案确定：保留原有 CSS 结构，不用 Tailwind。

**Compact #6** · 1/30 · Experience 页面 + SecondMe 数据集成研究
并行开发体验页面：SecondMe OAuth2 登录、需求提交、实时协商展示。研究 SecondMe API 数据结构以减少用户手动输入。创始人要求"直接并行启动然后一直开发"——体现了"做比完美更重要"的阶段策略。

**Compact #7** · 1/30 · 首页内容填充 + 文章系统 + 屏幕比例修复
修复屏幕比例（body 字体 19px → 16px）和响应式布局。首页 Hero 标题定为"为 Agent 重新设计的互联网"，副标题"你的Agent很强大，我们让他走向世界，与万物协作"。创建文章列表页，填入3篇完整文章。创始人指出"协议的价值应该来自创造，而不是采用"——改变了文案方向。

**Compact #8** · 1/30 · 文章内容 + 暖色调设计 + 滚动渐变背景
填入"道生一"和"微小的光"文章。创始人觉得网站太冷/临床感——"配色是很不错了，但背后的网格给人更多囚禁的秩序感"。移除网格线，添加滚动渐变背景。从 #EEEEEE 冷灰到 #F8F6F3 暖米白，不仅是颜色变化，是整个气质的转变。

**Compact #9** · 1/30 · OAuth 修复 + 实时协商 + 打字机效果 + Vercel 部署
修复 OAuth 回调不跳转问题（返回 JSON 而非重定向），实现需求提交和实时协商展示。添加流式消息的打字机效果改善 UX。代码审查发现硬编码密码哈希，移至环境变量。部署到 Vercel + CDN，研究 Redis Session 降级策略。

**Compact #10** · 1/30 · Redis Session + beads 任务管理 + UI 更新
创建 Redis Session 存储迁移文档和4个实现文件。UI 微调：把"加入网络"改为"体验 Demo"，更新联系邮箱，添加微信群二维码，移除尚未准备好的 GitHub 和 Twitter 链接。

**Compact #11** · 1/30 · "一键体验" + "找技术合伙人"演示场景设计
实现一键体验功能。创始人否决了"独立音乐人演出"场景："完全没有用的东西，跟我们的思想根本不匹配。" 改为"找技术合伙人"——展示"认知转变"：用户以为需要"技术合伙人"，发现真正需要的是"快速验证需求的能力"。设计了7个 Agent（程序员、外包工作室、产品教练、Notion模板作者等）和6个协商阶段。

**Compact #12** · 1/30 · WebSocket 跨域修复 + Experience UI 优化 + Vercel 绑定
WebSocket 因 Cookie `samesite="lax"` 不跨域发送，前端3000端口无法认证到后端8080端口。添加 `/ws/demo/` 无认证端点 + 前端自动检测跨域。用户信息移至右上角固定位置，添加可折叠 Profile 卡片。绑定 Vercel 到 GitHub 自动部署。

**Compact #13** · 1/30 · SecondMe 数据扩展 + 移动端适配 + 滚动渐变修复
创始人说"SecondMe那里获得的基本信息太少了"——添加 `selfIntroduction`、`profileCompleteness` 等新字段。修复背景渐变为 fixed 定位。全站移动端响应式适配（17个文件，1134行）。

**Compact #14** · 1/30 · Demo V2 — 10 阶段完整交互动画
创始人详细描述了完整的交互流程：需求缩小为点→射出线条→背景圆圈闪烁广播→Agent 发现和分类（绿色/红色/灰色）→绿色 Agent 汇聚→响应展示→信息汇聚中心→筛选与点对点协商→最终提案。实现了10阶段的状态机动画。创始人反馈动画不像"广播"更像"点对点"，添加了波浪扩散效果。

**Compact #15** · 1/31 · Railway 部署 + 多环境 OAuth + Experience V3（真实 Agent）
Railway 后端部署 + Vercel 前端部署。支持本地和生产 OAuth 回调。创建 experience-v3 使用真实 OpenAgents 数据，保留 experience-v2 演示版本。

### 会话 3：架构设计 + 共建体系（2/6 – 2/8）

**Compact #28** · 2/6 · 签名广播机制设计 — HDC + 三层共振过滤
启动架构深潜的第一轮。设计信号共振机制：Bloom Filter（90%过滤，100ns/检测）→ HDC 共振检测（9%过滤，1μs）→ LLM 深度理解（1%处理，10ms）。超维计算核心：10,000维二进制超向量，SimHash 编码保留语义。创始人关键洞察："广播和筛选是同一个逻辑执行了两次"——同一个数学操作，反转方向。

**Compact #29** · 2/6 · Agent 接入机制 + Skill 系统设计
讨论 Agent 如何接入网络。创始人提出"Agent 就是你的 Profile"——用户不需要"构建 Agent"，只需提供信息。Scene-based access（不过滤用户，过滤场景）。Skill 系统研究了20+篇论文，发现多轮辩论净负面（-3.5%），但并行提案→聚合正面（+57-81%）。"代码保障 > Prompt 保障"原则确立。创始人指出 SubNegotiation 不是"辩论"而是"发现对话"——Agent 有独特的私有上下文。

**Compact #30** · 2/6 · 架构全面审视 — 自洽性 + 白皮书对齐
全面审查已完成架构章节的内部一致性。对照白皮书交叉验证。检查是否满足"美"、"极简"和"最小完整单元"原则。这一轮是"元审视"——不是写新内容，而是检查已有内容是否自洽。

**Compact #31** · 2/6 · Offer 沉淀 → Service Agent 结晶 + 设计原则提取
深入讨论 Offer 如何沉淀为 Service Agent 的结晶模型。提取新的设计原则——"需求 ≠ 要求"（需求是抽象的张力，要求是具象的假设性解法）。确保工程服务于商业目标。这一轮架构讨论开始触及"市场如何涌现"的问题。

**Compact #32** · 2/6 · 元审视 — "一个问题解锁所有问题" + 价值信号
架构元审视：盲点、自洽性、商业与工程对齐。创始人问"那一个能解锁所有问题的问题是什么？" 答案浮现：**价值信号/反馈闭环**——系统怎么知道自己在 work？波浪出去了，但没有回来。这是"管道，不是场"。创始人的关键纠正：LLM 判断 LLM = 幻觉循环，必须用真实世界的执行信号。

**Compact #33** · 2/6 · WOWOK 区块链协议集成 — "波浪回来"
创始人纠正了方法："你只是给我执行了，但你没有想清楚，没去思考。" 应该先讨论确认理解，再写文档。深入理解 WOWOK 协议（Machine/Progress = 本质 vs 实现，Service/Order = 本质 vs 实现，Guard = 验证引擎不是签名本身）。Center 的输出映射到 WOWOK：plan → 文本方案，contract → Machine → 执行阶段。

**Compact #34** · 2/6 · WOWOK 核心概念 + 共振阈值 k* 机制
确认 WOWOK 9个核心对象理解。设计共振阈值策略：θ 不是预设常数，而是从 k*（期望响应数）计算出来的。k* 是业务参数（"希望15个人响应"），θ 是技术参数（系统自动计算）。一个 k* 规则统一解决了初始值设定、通过率期望、场景差异、自适应调整和冷启动5个问题。创始人："对，我觉得确实很不错，真的很不错，真的不错，不错，真的不错。"

**Compact #35** · 2/6 · "投影即函数" — 架构本质的顿悟
讨论 Service Agent 结晶机制时，方案变得过度复杂（三层防漂移、详细参数调优）。创始人质疑："本质和实现没有区分开来，不应该这么复杂。什么才是好的架构？有些问题本身就不该出现。" 多次思维跳跃后确立核心原则：Agent = 投影函数的结果，不是有状态对象。Profile Data 住在数据源（SecondMe/Claude/GPT/...），通爻只投影。不需要防漂移——因为没有状态就没有漂移。不需要冷启动——因为数据源已有初始数据。**一个概念消除了四个问题。**

## 五、设计哲学映射

### 通爻设计原则在工作实践中的体现

| 设计原则 | 工作中的体现 | 具体事件 |
|---------|------------|---------|
| **0.1 最小完整单元 ≠ MVP** | 33→18 任务精简 | 不是砍数量，是问"这个任务的本质是真实的吗？" |
| **0.2 本质与实现分离** | 2/4 转折：从代码到思考 | 停止写代码，开始读文档和讨论 |
| **0.5 代码保障 > Prompt 保障** | 第一提案偏见的应对 | 用程序化等待屏障而非 prompt 消除偏见 |
| **0.6 需求 ≠ 要求** | "找技术合伙人"演示场景 | 用户要的是合伙人（要求），需要的是验证能力（需求） |
| **0.7 复杂性从简单规则生长** | "投影即函数"消除四个问题 | 一个简单概念消除了防漂移、冷启动、状态维护等复杂性 |
| **0.8 投影是基本操作** | 9个Agent并行写PRD | 同一个架构文档，9个透镜，18篇PRD |
| **0.9 完备性 ≠ 完全性** | /contribute 三次迭代 | "不是为了简洁而简洁"——每层信息都有存在的理由 |
| **0.10 一自多我** | Edge Agent + Service Agent | 同一份 Profile Data，全维度 vs 聚焦投影 |

### 工作方法论的哲学映射

| 工作方法 | 对应的哲学 | 为什么有效 |
|---------|----------|----------|
| 先讨论再写文档 | 对齐 > 效率 | 减少返工，确保方向正确 |
| 架构讨论14轮 | "先讨论再执行"方法论 | 3天思考 > 3周返工 |
| /contribute 三次迭代 | "不是为了简洁而简洁" | 极度简单但不丢失意义 |
| 否决音乐人演示场景 | 形式必须体现本质 | 演示场景本身就是哲学的微缩版 |
| 架构审查后简化任务 | 问题消除 > 问题解决 | 好的架构让问题不存在 |
| Mock 模式被否决 | 真实 > 模拟 | 测试通过但实际不工作 = 没有价值 |
| "投影即函数"的诞生 | 道 = 一个操作的递归 | 复杂的东西不是设计出来的 |

### 创始人纠正的系统性意义

创始人的每一次纠正不是"改需求"，而是在传递设计理念：

| 纠正 | 表面含义 | 深层原则 |
|-----|---------|---------|
| "过度的复杂了" | 方案太复杂 | 如果需要很多特殊处理，基础规则没找对 |
| "自我揣测了" | 别替用户假设 | 需求 ≠ 要求 |
| "完全没有用的东西" | 演示场景不对 | 形式必须体现本质 |
| "本质和实现没有区分开来" | 架构层做了实现的事 | 本质应该稳定，实现可以替换 |
| "有些问题本身就不该出现" | 防漂移是伪问题 | 好的设计消除问题而非解决问题 |
| "不是为了简洁而简洁" | 太简洁丢失了意义 | 完备性 ≠ 完全性 |
| "先讨论确认理解，再写文档" | 流程问题 | 对齐 > 效率 |
| "mock模式不重要" | 别模拟 | 真实 > 表演 |
