## 核心命题：协议本身不赚钱

### 先例分析

| 协议/平台 | 协议层 | 钱在哪 | 关键洞察 |
|-----------|--------|--------|---------|
| TCP/IP | 完全免费开放 | ISP、SaaS、云服务 | 协议创造了互联网，互联网上的一切都在赚钱 |
| HTTP/HTML | 完全免费开放 | 网站、电商、广告 | Web 协议创造了万亿市场，协议本身一分钱没收 |
| Ethereum | 协议开源，有原生 token | Gas 费 + DeFi 生态 | Token 让协议本身可以捕获价值 |
| OpenAI | 论文公开，模型权重曾开放 | API 调用收费 | 不卖协议，卖最好的实现 |
| Stripe | 支付协议是开放标准 | 开发者友好的托管实现 | 协议开放，实现收费 |
| Linux | 完全开源 | Red Hat 卖企业服务 | 代码免费，服务收费 |

**一个关键区分**：

```
协议（语义、规则、数据格式）→ 应该开放、免费、越多人用越好
实现（托管平台、优化算法、规模化运维）→ 可以收费
产品（基于协议构建的具体应用）→ 可以大量赚钱
```

Stripe 不拥有"支付协议"，但它拥有**最好的支付实现**——开发者 10 行代码接入支付，这个便利性值 2.9% 的抽成。

OpenAI 不拥有"Transformer 论文"，但它拥有**最好的模型实现**——GPT-4 的质量和可用性值每百万 token $10。

**通爻的位置**：我们拥有的是 **Agent 协作协议**——如何让智能体发现彼此、协商方案、创造价值。协议本身应该开放。但最好的实现、最丰富的生态、最成熟的产品，这些是商业价值的来源。

### 为什么协议应该开放

**创始人的核心判断**：

> "很多公司的商业模式的根基是基于限制用户的——限制用户的数据、限制用户的选择、限制用户接触开放网络。这样的模式走不远。"

这不是理想主义，这是务实的商业判断：

1. **网络效应需要开放**：协议的壁垒是网络效应。封闭协议 → 用户数量受限 → 网络效应弱 → 容易被替代。开放协议 → 所有人都能接入 → 网络效应最大化 → 成为事实标准。

2. **大厂为什么做不了这个**：大厂的商业模式建立在数据壁垒之上。让用户拥有自己的数据、让 Agent 自由协作、让投影是端侧的——这从根本上动摇了大厂的护城河。**不是"大厂会不会做"的问题，是"大厂的商业模式不允许它们这么做"。**

3. **开源 ≠ 没有壁垒**：Linux 开源了 30 年，Red Hat 市值百亿。壁垒不在代码，在生态、品牌、社区、实现质量、运维经验。

## 产品赋能引擎：一套技术，N 个产品

### 通爻能做什么

通爻不是一个"产品"，是一个**能力**——"让一群人/Agent 快速发现彼此并协作"的能力。任何涉及"匹配+协作"的场景都能用。

### 具体的产品方向

#### 1. 黑客松团队组建

**现有痛点**：
- 主办方：参赛者手动组队，质量参差不齐，很多人找不到队
- 参赛者：发帖找队友、翻群聊、碰运气。有技能的人被埋没

**通爻提供的**：
- 参赛者填写 Template → Agent 注册 → 需求广播 → 共振匹配
- 不只匹配"你要什么"，还发现"你不知道你需要什么"
- 例：你以为你需要"前端开发"，但共振发现一个"交互设计师"跟你的项目方向高度互补——这是搜索范式找不到的

**商业模式**：
- 为主办方提供"智能组队"服务
- 主办方为更好的参赛体验付费
- 或者：为黑客松平台提供 API，按使用量收费

**与 SecondMe 的协同**：SecondMe 正在办 A2A 黑客松。完美的第一个场景——SecondMe 提供参赛者的 Profile 数据，通爻提供匹配和协商能力。

#### 2. 创业社区：找联创 / 找合伙人

**现有痛点**：
- 创始人花 80% 的时间在社交上——参加活动、发帖、聊天、筛人
- 找到"对的人"靠运气，而不是靠系统
- 最有价值的匹配往往是"意外的"——但意外靠不了搜索

**通爻提供的**：
- 创始人表达需求（可能是模糊的："我需要有人帮我把想法变成产品"）
- 通爻理解"需求 ≠ 要求"——不按"985毕业"筛选，而是按真正的能力互补匹配
- Center Agent 综合多个候选人，甚至建议"你不需要一个联创，你需要的是验证能力"

**商业模式**：
- 社区 / 孵化器 为更好的匹配服务付费
- 或者：创业者为 premium 匹配付费（类似 LinkedIn Premium）

#### 3. 企业内部资源协调

**现有痛点**：
- 大公司里不知道隔壁部门有人能帮忙
- 内部需求通过层层汇报传递，效率低下
- 跨部门协作需要"认识对的人"

**通爻提供的**：
- 员工的技能/项目/职责编码为 Agent Profile
- 需求广播到内部网络 → 相关的人自动被发现
- 不需要"认识谁"，需要你的人会自己出现

**商业模式**：企业 SaaS，按座位或按使用量收费

#### 4. 学术/研究合作

**现有痛点**：
- 跨学科合作极难发起——你怎么知道物理系有个人在做的东西跟你的生物学研究相关？
- 现有平台（ResearchGate、Google Scholar）是搜索范式——你必须知道你在找什么

**通爻提供的**：
- 研究者的 Profile 编码为 HDC 向量
- HDC 的跨域干涉天然适合发现跨学科关联——这是这个技术最独特的价值
- 一个生物学家发出"我需要模拟蛋白质折叠"的需求 → 一个物理学家的"分子动力学模拟"经验产生共振 → 意外合作

**商业模式**：大学/研究机构付费，或与学术平台合作

#### 5. 自由职业 / 服务市场

**现有痛点**：
- Upwork/Fiverr 是搜索范式——客户搜索"React开发者"，得到 10000 个结果
- 无法发现"你不知道你需要的技能"
- 价格竞争导致质量下降

**通爻提供的**：
- 响应范式：客户发出需求 → 能做的人自己来
- 不按技能标签匹配，按真实能力共振
- Center 发现互补组合："你以为你只需要一个前端，但加一个 UX 设计师你的项目会好 10 倍"

## 个人数据主权：比通爻更大的事

### 问题的本质

当前世界：你的数据散落在 100+ 个平台，每个平台各存一份"你的影子"。

- LinkedIn 知道你的职业经历
- SecondMe 知道你的性格和价值观
- GitHub 知道你的代码能力
- 微信知道你的社交关系
- 豆瓣知道你的审美偏好
- 抖音知道你的注意力模式

没有任何一个平台有"完整的你"。更关键的是——**你不拥有这些数据**。平台拥有。

隐私策略是让人头疼的权宜之计——GDPR 要求平台"让你删除数据"，但根本问题不在于"能不能删"，而在于**数据一开始就不应该在平台手里**。

### 范式转变

```
当前：
    你 → 注册平台 A → 平台 A 存储你的数据 → 你想走？数据留下
    你 → 注册平台 B → 平台 B 存储你的数据 → 数据孤岛
    你 → 注册平台 C → ……
    → 100 个平台，100 份残缺的"你"，你一份都不拥有

未来：
    你 → 你的个人数据层（你拥有）
    平台 A → 申请读取权限 → 你授权透镜 A（只看职业维度）
    平台 B → 申请读取权限 → 你授权透镜 B（只看兴趣维度）
    平台 C → ……
    → 1 份完整的"你"，你完全拥有，平台只是"借看"
```

### 与通爻架构的深层关联

这不是两个不相关的项目——**个人数据层就是 ProfileDataSource 的实体化**。

通爻的 ProfileDataSource 接口已经定义了：
- `get_profile(user_id)` → 读取用户数据
- `update_profile(user_id, experience_data)` → 协作数据回流
- Adapter 架构 → 可插拔的数据源

如果有一个统一的个人数据层：
- 不需要 N 个 Adapter（SecondMe、Claude、Template...），只需要 1 个：指向用户的个人数据层
- 投影更完整——因为数据源是完整的"你"，不是某个平台上残缺的"你"
- 数据所有权天然解决——用户的个人数据层在用户手中

**透镜即授权**：用户对每个应用授权一个"透镜"——这个应用只能看到你数据的某些维度。这跟 Service Agent 的透镜是**同一个概念**。架构文档中的 lens 参数，在个人数据层的语境中就是 access scope。

```python
# 通爻的透镜
edge_vector = project(profile_data, lens="full_dimension")
service_vector = project(profile_data, lens="focus_on_frontend")

# 个人数据层的授权透镜
towow_view = personal_data.authorize(app="towow", scope="full")
linkedin_view = personal_data.authorize(app="linkedin", scope="career")
dating_view = personal_data.authorize(app="dating", scope="personality+interests")
```

同一个操作。同一个数学。

### 与 SecondMe 的关系

SecondMe 做的是"通过社交获取关于社交的数据"——这是一个非常好的起点，但它天然受限于社交维度。

创始人的洞察：
> "通过社交来获取关于社交的数据，终究是没有办法获取所有的数据。但你有一个完全的个人数据层，这是完全不一样的东西。"

SecondMe 可以是个人数据层的一个**数据贡献者**——它贡献社交维度的数据。但个人数据层还包括：
- 职业数据（来自工作经历）
- 技能数据（来自项目、代码、作品）
- 健康数据（来自可穿戴设备）
- 消费数据（来自购买记录）
- 学习数据（来自课程、阅读）
- ……

个人数据层不是 SecondMe 的竞争者，而是 SecondMe 的**超集**。SecondMe 是最好的社交维度数据源，个人数据层是所有维度的整合。

### 技术方向（初步思考）

| 方向 | 核心思路 | 优势 | 挑战 |
|------|---------|------|------|
| 链上身份 + 链下存储 | 区块链管权限和身份，实际数据在 IPFS/Arweave 或本地 | 去中心化、不可篡改 | 性能、用户体验 |
| 端侧硬件 | 个人设备（手机/个人服务器）持有数据，应用通过 API 读取 | 物理安全、完全控制 | 设备丢失/损坏 |
| 混合方案 | 链上身份+权限，端侧存储，加密云备份 | 兼顾安全和可用 | 复杂度 |

已有探索：
- **Solid**（Tim Berners-Lee）：Web 去中心化，用户数据存在自己的 Pod
- **DID/可验证凭证**：W3C 标准，去中心化身份
- **Ceramic Network**：去中心化数据流协议
- **IPFS/Arweave**：去中心化存储

这些都是好的技术基础。但没有人把它们组合成一个**用户友好的、面向普通人的个人数据层**。

## 生态飞轮

```
通爻协议（核心引擎）
    ↓ 赋能
垂直产品（黑客松、企业、社区）
    ↓ 吸引
用户和开发者
    ↓ 产生
协作数据（回声信号）
    ↓ 沉淀到
个人数据层（用户拥有）
    ↓ 反哺
通爻投影更精准（因为数据源更完整）
    ↓ 提升
产品效果更好
    ↓ 循环
更多人参与 → 更多数据 → 更好匹配 → ……
```

**飞轮的关键**：数据不是被平台"提取"的，而是"沉淀"在用户自己手中的。用户越用通爻，自己的数据越丰富，自己的未来协作越精准。**激励是内生的，不需要外部奖励机制。**

这跟传统平台的飞轮根本不同：
- 传统：用户数据 → 平台变强 → 用户被锁定 → 平台收税
- 通爻：用户数据 → 用户变强 → 用户更愿意参与 → 网络变强

## 大厂为什么做不了

这不是技术问题，是商业模式冲突。

**大厂的护城河是什么？** 数据壁垒。

- 微信的护城河：你的社交关系在微信里，你走不了
- 抖音的护城河：你的兴趣画像在抖音的算法里，别的平台推不准
- LinkedIn 的护城河：你的职业网络在 LinkedIn 里，你不敢删

**通爻/个人数据层要做的事情**：让用户拥有自己的数据，让数据可以跨平台流动。

这直接动摇了大厂的根基。它们不是"不会做"，是"不能做"——做了就是自毁护城河。

类比：Kodak 发明了数码相机，但不会去推广它，因为数码相机会摧毁它的胶卷业务。

**这就是通爻的机会**：做大厂不能做的事。

### 反驳"大厂会抄"

> "你们的技术大厂也能做啊。"

回应：
1. **能做 ≠ 会做**。大厂的商业模式不允许它们做"让用户拥有数据"的产品。
2. **协议的壁垒不在技术，在网络效应**。先到者定义标准。TCP/IP 的技术不复杂，但一旦成为标准就不可替代。
3. **开源 + 社区 = 防御**。协议开源后，社区参与共建，不是一家公司可以复制的。
4. **生态 > 技术**。100 个开发者在你的协议上构建产品，这个生态是抄不走的。

## V1 → 商业化路径

### Phase 1：证明机制（当前）

- 资金：天使轮/种子轮
- 成本：人才为主，算力可忽略
- 目标：在 1-2 个场景中证明"响应范式 > 搜索范式"
- 收入：零。全力验证。

### Phase 2：场景验证

- 与 SecondMe 合作 A2A 黑客松 → 第一个真实场景
- 与 1-2 个创业社区合作 → 验证匹配质量
- 产出：案例数据、用户反馈、产品形态
- 收入：可能有小量服务费，但不是重点

### Phase 3：产品化

- 基于验证结果，打磨 1-2 个垂直产品
- 开放 API / SDK，让开发者在协议上构建
- 收入：服务费、API 调用费、企业订阅

### Phase 4：协议普及

- 从平台模式逐步切换到协议模式
- 端侧自付费（Agent 为自己的投影和 Offer 生成买单）
- 个人数据层项目启动
- 收入：协议生态中的增值服务、企业版、数据层产品

### Phase 5：生态成熟

- 协议成为 Agent 协作的事实标准
- 第三方产品大量涌现
- 通爻团队的角色：协议维护者 + 核心产品开发者 + 生态运营者
- 收入：多元化（产品、服务、企业、生态分成）

## 产品先行，协议涌现

### 核心判断：通爻不是"协议公司"

没有成功的"协议公司"。所有看起来像"协议成功"的案例，本质上都是"产品成功后协议被开放"：

- TCP/IP 是 DARPA 资助的研究项目。商业化是 Cisco、ISP——产品。
- HTTP 是 CERN 做的。商业化是 Netscape、Apache——产品。
- Ethereum Foundation 是非营利。赚钱的是 Uniswap、MetaMask——产品。
- OpenAI 不拥有 Transformer 论文，但拥有最好的实现——产品公司。

**正确定位**：做出 Agent 协作最好的产品，然后把协议开放出来。不是"先做协议，再看谁来做产品"。

**顺序很重要**：没有产品验证的协议 = 空中楼阁。Stripe 没有先"发布支付协议"然后等人用，而是做了一个产品（10 行代码接入支付），产品成功后 API 标准事实上成了"协议"。

通爻的路径：先做出杀手级产品 → 其他场景看到效果想用 → 开放协议 → 生态自然生长。

### 冷启动策略：两条腿走路

1. **SecondMe 已有数据** — 利用 SecondMe 的社交数据作为 Profile 种子，不是"所有数据"，但比空白 Profile 好得多。
2. **场景驱动** — 一个一个线下场景跑出来：黑客松、创业活动、企业内部协调。每个场景的协作回声沉淀为数据，丰富下一个场景的投影质量。

两条路并行：SecondMe 提供初始数据丰富度，场景产出增量数据。飞轮从小开始转。

## SecondMe A2A 黑客松：竞争分析与定位 <!-- TODO: 黑客松参赛策略待详细讨论——Demo 设计、评委角度、具体执行计划 -->

### 其他参赛项目的七个结构性缺陷

SecondMe 黑客松上百个团队在做 A2A 应用——AI 招聘、Agent 相亲、Agent 这个那个。本质上：都在用 Agent 重建互联网上已有的垂直应用。

| # | 问题 | 本质 | 通爻怎么解 |
|---|------|------|----------|
| 1 | 没有数据回流 | 一次性使用，协作结果不回到 SecondMe，无法持续进化 | Echo 机制 → 执行结果通过 WOWOK 回流 → Profile 持续丰富 |
| 2 | 数据不统一 | 每个应用各存一份碎片数据 | ProfileDataSource + 透镜授权 = 一份数据，多个视图 |
| 3 | 服务器瓶颈 | SecondMe 成为所有应用的中心化瓶颈 | 端侧计算 + 签名广播，协议层不依赖单一服务器 |
| 4 | **搜索范式** | 用更聪明的 Agent 做搜索，没跳出搜索的框架 | **响应范式**——Agent 来"应"，不是用 Agent 去"找" |
| 5 | 无协调层 | 每个项目自己写 ad hoc 的多 Agent 协调 | Center Agent + Negotiation Unit（可递归、可组合） |
| 6 | 不可组合 | 垂直孤岛，招聘 Agent 和相亲 Agent 各自封闭 | 协商单元可组合，同一个需求触发多维度匹配 |
| 7 | 质量不可验证 | LLM 推荐 LLM 评估 = 幻觉循环 | 链上执行回声 = 真实结果反馈 |

**最致命的是 #4**。其他项目只是"用 Agent 包装搜索"。通爻做的是不同的范式——**你不搜索答案，答案来找你**。而且答案里可能包含你没想到的东西。

### 黑客松策略

**不做"又一个垂直应用"**。评委见了 47 个"AI 匹配"项目之后看到第 48 个不会兴奋。

**做的是**：展示所有其他项目都缺失的那一层。

**Demo 设计**：
- 用黑客松本身作为场景
- 参赛者说"我要找前端开发" → 不搜索，广播需求
- 三个 Agent 共振响应。但还发现一个 UX 研究者——她的认知科学背景跟这个 AI 教育项目意外互补（搜索找不到这个）
- Center 综合 Offer，发现缺口：可能需要了解教育政策的人 → 自动发起子协商
- 协作结果回流 → 下次推荐更精准

**关键一幕**：如果能在 demo 里造出一个"我没想到他能帮到我"的时刻——这比任何 PPT 都有说服力。这就是响应范式的核心价值。

**评委看到的差异**：别人做"更好的搜索"，你做"超越搜索的发现"。别人是一次性匹配，你有数据闭环。别人做一个垂直场景，你现场演示同一个引擎在另一个场景也能工作。

## 投资叙事

### 一句话

> "A2A 告诉 Agent 怎么说话。通爻告诉 Agent 该找谁说、该说什么。"

### 种子轮 / 天使轮

投资人买的是团队 + 方向 + 时机。

**方向**：Agent 之间如何协作是当下最大的空白。A2A 只是通信协议（TCP），不是协作协议（WWW）。通爻做的是 Agent 协作的 WWW。

**时机**：SecondMe 黑客松上百个团队证明市场需求已经存在，但所有团队都遇到同样的结构性问题——正是通爻解决的问题。

**常见 Q&A**：

| 投资人的问题 | 回答 |
|------------|------|
| 大厂会不会做？ | 大厂的商业模式建立在数据壁垒上。Agent 协作的前提是数据跨平台流动——直接瓦解大厂护城河。不是不会做，是商业模式不允许做。柯达发明了数码相机但不推广它。 |
| A2A 本身不够吗？ | A2A 是传输层（TCP），还需要应用层。怎么发现、匹配、协商、验证——TCP 上面需要 HTTP/DNS/TLS，A2A 上面需要通爻。 |
| 怎么赚钱？ | 协议开放，产品赚钱。Stripe 不拥有支付协议，但拥有最好的支付实现——10 行代码接入支付值 2.9% 抽成。我们做 Agent 协作的 Stripe。 |
| 壁垒在哪？ | 网络效应 + 数据飞轮。第一个达到临界规模的 Agent 协作网络，后来者的迁移成本是指数级的。而且飞轮是用户拥有的——越用越不想离开。 |
| 竞争对手？ | 直接竞品几乎没有。间接竞品是垂直应用（AI 招聘等），但它们是搜索范式，不在同一维度。最近的是 SecondMe 自己做协作层，但 SecondMe 的核心是数据源，我们互补而非竞争。 |

### A 轮需要的证据

到后续轮次，需要可量化的数据：
- 黑客松匹配质量对比（响应范式 vs 搜索范式，意外发现率）
- 用户留存与 Profile 丰富度的相关性
- 开发者生态（多少第三方在协议上构建）
- 协商单元跨场景复用率

## 总结：通爻的经济叙事

**一句话**：通爻做的是"Agent 协作的 TCP/IP"——协议开放，生态繁荣，价值从协作中涌现。

**三段话**：
1. 在一个有百亿 AI Agent 的世界里，它们需要一种方式来发现彼此、协商方案、创造价值。通爻提供的是这个基础协议——响应范式，而不是搜索范式。
2. 协议本身不收费，就像 TCP/IP 不收费。但基于协议的产品可以创造巨大价值——黑客松匹配、企业协作、创业者连接、学术合作、自由职业市场——每一个都是独立的商业机会。
3. 更深层的是，通爻的投影架构天然支持个人数据主权——用户拥有自己的数据，应用只是"借看"。这不只是隐私保护，而是一种新的互联网范式——从"平台拥有你的数据"到"你拥有你的数据"。
